# BullMQ for Jobs + Events — Understanding & Plan

**Date:** 2026-06-15  
**Source:** 7-agent understanding workflow (wf_9dee4dfa-51f) + lead deep-read of both subsystems  
**Goal:** Fully functional BullMQ backend for the jobs AND events subsystems — jobs + event streams on BullMQ for automation, scheduling, and jobs in general.

## Current state

JOBS on BullMQ are functionally implemented behind the core IJobOrchestrator contract but NOT actually reachable end-to-end, and have silent semantic gaps. BullMQJobOrchestrator (runtime/subsystems/jobs/job-orchestrator.bullmq-backend.ts) extends DrizzleJobOrchestrator — Postgres job_run stays the domain source of truth (dedupe, concurrency-collision short-circuit, parent/root, cancel-cascade, replay, upsertJobRows) and start/cancel/replay layer queue.add/job.remove on top. BullMQJobWorker (job-worker.bullmq-backend.ts) is a real consumer: one Worker per pool, native claim, loads job_run, marks running, resolves the handler via ModuleRef, runs the same JobContext (ctx.step/ctx.spawnChild), writes terminal status, mirrors final failure on the `failed` event. Module wiring (JobsDomainModule/JobWorkerModule bullmq branches), lazy `bullmq` peer-dep loading, queue-name resolution, and connection teardown are complete. CRITICAL CONTRADICTION found during verification: the CLI subsystem descriptor in src/cli/shared/subsystem-detect.ts:79-83 lists jobs `backends: ['drizzle','memory']` — so `subsystem install jobs --backend bullmq` is REJECTED by isValidBackend (subsystem.ts:207-214) with exit 2. The shipped BullMQ jobs backend is therefore un-installable via the CLI today; it only works for a hand-edited barrel/forRoot call. The backendFileFilter `.bullmq-backend.ts` suffix pruning (subsystem.ts:291) is correct and subsystem-agnostic, but the descriptor gate blocks it.

EVENTS on BullMQ DO NOT EXIST. runtime/subsystems/events ships drizzle (outbox, default), memory (test), and redis (ioredis Pub/Sub) backends only. The events module backend union is 'drizzle'|'memory'|'redis' (events.module.ts:99) with no bullmq arm; the events descriptor lists only ['drizzle','memory'] so even the existing Redis backend is un-installable via CLI. RedisEventBus.findById returns null (event-bus.redis-backend.ts:181), publish ignores tx (non-transactional), and it is explicitly bridge- and scheduler-incompatible.

SCHEDULING/AUTOMATION is Drizzle/Postgres-outbox-bound. Recurring cadence is event-shaped (ADR-039): an event YAML `schedule:` block is materialized by a single in-process EventScheduler setInterval loop that INSERTs one payload-free domain_events row per (type, slot) idempotently (ON CONFLICT on a partial UNIQUE slot-key index); the bridge then spawns wrapper+user runs. The EventScheduler is drizzle/memory-only (skipped under redis). Jobs have NO native cron/repeatable — only one-shot StartOptions.runAt, and the BullMQ dispatch() builds jobOpts from retry+dedupe ONLY (line 225), never reading run.runAt, so even one-shot delay silently degrades to immediate dispatch on BullMQ. The event→job bridge (BridgeOutboxDrainHook) does trigger-match + bridge_delivery ledger + wrapper job_run insert INSIDE the Drizzle drain's per-event tx — there is no BullMQ/outbox-less equivalent. RFC-0005 cadence-on-job desugar (emitter #7/#8) is unbuilt.

NET: jobs run on BullMQ but only via hand-wiring and with runAt/priority/collisionMode:'queue' silently lost; events do not run on BullMQ at all; no broker round-trip (port-promotion gate) has ever executed in-repo (BULLMQ-1 §Verification is explicit); scheduling stays Postgres-outbox-only.

## Design recommendation (decided)

Adopt the dealbrain audit's core conclusion and ADR-039's locked verdict rather than 'put events on BullMQ to match jobs'. The architectural end state is SEPARATION on one Redis: a narrow durable IEventBus + a BullMQ-orchestrated IJobOrchestrator + the bridge as the seam — NOT 'events and jobs are the same BullMQ thing'.

EVENTS BACKEND — durable, not Pub/Sub. The existing RedisEventBus is fire-and-forget Pub/Sub: findById->null, tx ignored, bridge/scheduler-incompatible. Per no-backwards-compat, do NOT add a third Redis variant beside it; the clean move is to BUILD event-bus.bullmq-backend.ts as a DURABLE event bus and DELETE the dead RedisEventBus (it is un-installable via CLI and structurally incompatible with the bridge). The BullMQ events backend keeps the SAME transactional-outbox spine the bridge relies on: domain_events remains the committed-event store and findById source of truth, so the bridge's single-transaction idempotency (outbox insert + bridge_delivery UNIQUE(event_id,trigger_id) + wrapper job_run, all in one Postgres tx) is preserved verbatim. What BullMQ adds is the DISPATCH/wake half: publish writes the outbox row (in the caller's tx) and the drain enqueues to BullMQ instead of being polled. This mirrors the jobs design exactly (Postgres = source of truth, BullMQ = claim/dispatch) and means 'BullMQ events' is a dispatch-only backend over a Postgres outbox — not a Redis-resident event log. Core IEventBus stays narrow and unchanged (publish/publishMany/subscribe/findById); the materialize* methods stay optional extension methods.

BRIDGE — keep the seam, swap the wake. The bridge stays the jobs worker draining reserved events_* pools. The only change for an all-BullMQ stack is that the wrapper job_run insert is followed by an enqueue callback (orchestrator dispatch) rather than relying on a poll — the bridge_delivery ledger, the per-event tx, and Case-B pre-write dedup are untouched. This honors ADR-023's exactly-once invariant and avoids re-deriving idempotency from BullMQ jobId (which only covers job-side dedup, not the event->delivery pair).

SCHEDULING — implement ADR-039's committed mapping as a backend materializer swap, NOT a new core method. The YAML `schedule:` contract ({every, align, catchUp, maxCatchUpSlots}) is the spine and is identical across backends. The swappable part is the materializer: Drizzle = EventScheduler setInterval + ON CONFLICT slot insert; BullMQ = upsertJobScheduler (Job Scheduler) registered per scheduled-event type from the eventRegistry, with reconcile-on-boot = upsert-desired + prune-orphans (mandatory — closes the ENG-605 zombie-scheduler hole). To preserve the unified time->fact->bridge->job flow, the BullMQ Job Scheduler should produce the SAME scheduled DOMAIN EVENT into the outbox (a tiny framework repeatable job whose body is `eventBus.publish(scheduledTick)`), NOT fire the user job directly — this keeps cron, RFC-0005 cadence-on-job, and the bridge on one path and keeps job_run the source of truth. When the BullMQ scheduler is active, the EventScheduler setInterval loop and the slot-key UNIQUE index do not run (one source of truth). Cron strings stay deferred at the YAML layer (additive later: {cron}->{pattern}); do NOT add a cron parser dependency now. Do NOT add a cron/upsertScheduler method to IJobOrchestrator — that would force a Drizzle/Memory scheduler loop into the core and break portability; cadence stays event-shaped.

JOBS — close the core-contract leaks (these are bugs, not extensions). In dispatch(): map runAt->delay=max(0,runAt-now), set the BullMQ priority opt from StartOptions.priority, and implement a collisionMode:'queue' gate in the worker (release/re-enqueue when a same-concurrency-key run is active) so the three core StartOptions fields behave identically to Drizzle. Surface removeOnComplete/removeOnFail/lockDuration/stalled and Bull Board as opt-in jobs.extensions.bullmq.* config (consumer mounts Bull Board).

COMPOSE ON ONE REDIS. Reuse the jobs bullmq.config.ts connection/queue-prefix machinery for events (shared BULLMQ_CONNECTION semantics, queue_prefix namespacing so jobs+events share one Redis). Honor the optional-peer contract everywhere: backend files import type-only + await import('bullmq'); never re-export backend classes from the public barrels; rely on the `.bullmq-backend.ts` suffix filter. Tighten the events/jobs backend enums in the config schema and make both module backend switches throw on an unknown default.

VERIFICATION is the gating deliverable. Build the reusable cross-backend contract harness (the thing BULLMQ-1 said does not exist) + a testcontainers Redis(+Postgres) integration recipe out of test-all (Docker-gated), and run the same suite against drizzle/memory/bullmq for BOTH subsystems. Only flip defaults / declare 'fully functional' once that round-trip is green; until then both BullMQ backends stay opt-in (default drizzle), exactly as ADR-022 §claim requires.

## Gap list

- G1 (BLOCKER, jobs reachability): jobs descriptor backends=['drizzle','memory'] (subsystem-detect.ts:79-83) — add 'bullmq' so `subsystem install jobs --backend bullmq` is accepted. Without this the shipped BullMQ jobs backend is un-installable via the CLI; everything else for jobs is downstream of this one-line gate.
- G2 (jobs core-contract leak): BullMQ dispatch ignores run.runAt — map runAt to BullMQ delay = max(0, runAt - now) so one-shot scheduled/delayed starts work; runAt is a CORE StartOptions field and the backend currently violates it.
- G3 (jobs core-contract leak): priority is silently dropped — set the BullMQ `priority` job opt from StartOptions.priority/job_run.priority so claim ordering matches Drizzle.
- G4 (jobs core-contract leak): collisionMode:'queue' concurrency gating is a no-op under BullMQ — the Drizzle worker re-queues a claimed run when another run with the same concurrencyKey is running (job-worker.ts:629-652); BullMQJobWorker.process has no equivalent gate. Either implement a concurrency-key release gate or use BullMQ group/rate-limit; whichever, the 'queue' mode must not silently let both run.
- G5 (events backend missing): no event-bus.bullmq-backend.ts, no 'bullmq' arm in the events module backend union (events.module.ts:99), no loadBullMqEventBus lazy loader, no buildEventBusAsync branch, no connection token binding, no events composer arm in subsystem-barrel-generator.ts (events composer at 257-322 has no redis/bullmq arm).
- G6 (events installability): events descriptor backends=['drizzle','memory'] (subsystem-detect.ts:73-77) — add 'bullmq'. Decide separately whether to also add 'redis' or DELETE the dead RedisEventBus per no-backwards-compat.
- G7 (events durability/findById): a BullMQ events backend must satisfy IEventBus.findById returning the COMMITTED event (the bridge re-fetches at claim time) — Pub/Sub-style null is not acceptable for a backend that wants the bridge. This forces a durable design (BullMQ Queue / Redis stream / job_run-as-outbox), not fire-and-forget Pub/Sub.
- G8 (scheduling on BullMQ): no BullMQ scheduler materializer. ADR-039 already commits `schedule:` YAML -> BullMQ upsertJobScheduler (Job Scheduler) with reconcile-on-boot = upsert-desired + prune-orphans; the EventScheduler setInterval loop is drizzle/memory-only. Build the BullMQ scheduler arm that registers/prunes Job Schedulers from the eventRegistry schedule blocks.
- G9 (zombie-scheduler hazard): a broker-side scheduler re-introduces the zombie-scheduler bug class (ADR-039 ENG-605) the outbox model structurally avoids — the BullMQ scheduler MUST prune orphan Job Schedulers on boot (reconcile = upsert-desired + prune), and the slot-key UNIQUE index / lastScheduledSlotMs backfill machinery must NOT also run (one source of truth).
- G10 (bridge on BullMQ — the central seam): BridgeOutboxDrainHook trigger-match + bridge_delivery + wrapper job_run insert presupposes a Postgres outbox drain tx; on an all-BullMQ stack there is no drain to hook. Decide and implement the events->jobs activation path (recommendation: keep the durable outbox+bridge spine and make the wrapper-run insert enqueue to BullMQ via the orchestrator, rather than abandoning the bridge_delivery UNIQUE(event_id,trigger_id) ledger and same-tx idempotency).
- G11 (port-promotion gate never ran): the broker round-trip (start->queue.add->worker->completed/failed/retry/cancel-remove) has never executed in-repo for jobs, and would not for events either. Build a reusable cross-backend contract harness + a testcontainers-Redis (+ Postgres) integration recipe, gracefully skipped without Docker, parallel to the existing Postgres integration recipes (justfile:155-179). Until green, both BullMQ backends stay opt-in.
- G12 (Bull Board extension): config-only today (resolveBullMqConfig.bullBoard) — no generated mount/adapter/auth. Surface as an opt-in extension (consumer mounts), and document that scheduled jobs are visible there.
- G13 (memory parity): any NEW core scheduling capability (if cron is promoted to core) needs a Memory implementation too, or the cross-backend contract suite breaks across the three backends. Prefer keeping cron as an extension/event-shaped to avoid this.
- G14 (events schema skip): a brokered/durable-non-Postgres events backend ships no domain_events outbox schema; the subsystem-schema-generator + install flow must not emit the events outbox table for a bullmq events install (consumer would get a table they never write).
- G15 (dedupe re-enqueue invariant): the orchestrator's 'trust jobId idempotency' re-enqueue (lines 190-200) is an untested hand-wave — removeOnComplete interacting with re-add could resurrect a deduped job. Cover with a real broker test in the harness from G11.
- G16 (config validation): codegen-config schema is .passthrough(); a typo'd events/jobs backend falls through to a switch default. Tighten the jobs/events backend enum and ensure the module backend switch has an explicit throwing default.
- G17 (living docs): BULLMQ-1 §Extensions says cron out-of-scope and points at the wrong ADR; update it (and ADR-039) in the same PRs as the work lands — write the new BullMQ-events spec (BULLMQ-2) and the ADR-041 deciding the all-BullMQ topology.

## Open questions (RESOLVED by lead, autonomous)

- Events transport for BullMQ: durable dispatch over the Postgres outbox (recommended — preserves the bridge's same-tx exactly-once and findById, mirrors jobs) vs a Redis-resident durable log (BullMQ Queue / Redis Streams) that would require re-deriving the event->delivery idempotency ledger. The recommendation picks outbox-dispatch; confirm you do not want a Postgres-free events path.
- Delete RedisEventBus entirely (no-backwards-compat — it is un-installable via CLI, non-transactional, bridge/scheduler-incompatible) vs keep it as a fire-and-forget option. Recommendation: delete. Confirm no consumer depends on Pub/Sub semantics.
- BullMQ scheduler output: should the BullMQ Job Scheduler produce the scheduled DOMAIN EVENT into the outbox (recommended — keeps cron + RFC-0005 + bridge on one time->fact->job path, job_run stays source of truth) or fire the user job DIRECTLY via a BullMQ repeatable job (fewer hops, but bypasses the outbox/bridge and splits the activation model)? ADR-039 §159 says 'a repeatable producer of the same event', which favors the outbox-event approach — confirm.
- Cron string support timing: ADR-039 defers cron ({cron}->{pattern}) as additive. BullMQ supports it natively with zero new dependency. Ship cron now (BullMQ-only extension, Drizzle stays interval-only) or hold to the locked interval-only contract for backend parity? Recommendation: hold; ship interval-only first, add cron as a follow-up extension.
- collisionMode:'queue' implementation on BullMQ: a Postgres-checked release/re-enqueue gate (matches Drizzle semantics exactly) vs BullMQ-native group concurrency / rate-limiting (more idiomatic, slightly different semantics). This is a semantics call worth a human decision before BULLMQ-JOBS-3.
- Port-promotion gate location: run the broker round-trip in codegen-patterns CI via testcontainers Redis+Postgres (requires Docker-in-CI, which test-all currently avoids) vs keep it a local/consumer-repo gate as BULLMQ-1 chose. Recommendation: add the recipe here, Docker-gated and out of test-all, plus optionally a CI services block.

## Risks

- Default-flip is a silent behavior change: today runAt delay, priority ordering, and collisionMode:'queue' are honored by Drizzle but no-ops under BullMQ. JOBS-2/3 must land and be broker-verified BEFORE any consumer flips jobs.backend:bullmq, or scheduled/priority/serialized jobs misbehave silently. Per CLAUDE.md the core contract must be portable — these are leaks to close, not extensions.
- Moving the bridge onto BullMQ risks losing exactly-once: ADR-023's correctness rests on a single Postgres tx (outbox insert + bridge_delivery UNIQUE + wrapper run). BullMQ has no cross-store tx. The recommended outbox-dispatch design preserves this; any Redis-resident-log alternative must explicitly re-derive the event->delivery ledger (BullMQ jobId dedup covers job-side only).
- Zombie-scheduler regression: a broker-side Job Scheduler re-introduces the ENG-605 class the outbox model structurally avoided. Prune-orphans-on-boot is mandatory in BULLMQ-SCHED-1; running both the setInterval materializer/slot-key index AND the BullMQ scheduler would double-fire — exactly one must be the source of truth per backend.
- The whole BullMQ path is UNVERIFIED end-to-end (BULLMQ-1 §Verification): the broker round-trip has never run in-repo. Per the smoke-filter memory, do not mistake the green unit suite (wiring + pure helpers) for the gate. BULLMQ-VERIFY is the load-bearing deliverable; treat 'fully functional' as blocked on it.
- Memory-backend parity: if cron is ever promoted into the core contract it needs a Memory implementation or the cross-backend contract suite breaks. Keeping cadence event-shaped (extension/materializer swap, not a new core method) avoids this — do not bolt upsertScheduler onto IJobOrchestrator.
- Optional-peer footguns: a new event-bus.bullmq-backend.ts must use type-only bullmq imports + await import, must NOT be re-exported from events/index.ts, and relies on the .bullmq-backend.ts suffix filter — any slip drags the peer into drizzle consumers' tsc (TS2307 / smoke-subsystems findStaticPeerImports failure).
- config schema is .passthrough(): a typo'd events/jobs backend is not caught by Zod and falls through the module switch default — tighten the enum and add throwing defaults, or a misconfig silently picks the wrong backend.
- events-schema for non-Postgres variants: the recommended design KEEPS the Postgres outbox, so the domain_events schema stays; if a future Postgres-free events variant is chosen, the schema-generator/install must stop emitting the outbox table or consumers get a dead table (G14).
- Doc drift: ADR-040 is taken (ADR-041 is the next free number); BULLMQ-1's cron-out-of-scope pointer is stale (points at ADR-025, superseded by ADR-039). Update these in the same PRs per the living-docs principle, or every future agent pays the drift.
