## Problem

`SubsystemInstallCommand.executePackageMode` (`src/cli/commands/subsystem.ts:382â†’778`) reduces a package-mode install to: record in `subsystems.install` â†’ inject config block â†’ regenerate the two barrels. It returns before `runJobsScaffold`, so the three jobs templates (`worker.ejs.t`, `main-hook.ejs.t`, `job-orchestration.schema.ejs.t`) are never rendered. Package mode is the DEFAULT (ADR-037) â€” so the #516-fixed worker, whose package-import branch (`@pattern-stack/codegen/runtime/subsystems/jobs/index`) is fully unit-tested, is unreachable in the mode it was built for.

The short-circuit was correct for what it excluded at the time: vendored runtime copies, vendored schema templates, `.gitkeep`s. It over-excluded the **consumer-owned** files.

## Post-implementation (discovered during build â€” living-docs)

The pre-implementation spec above is the intent; this section is the
post-implementation truth (per the living-docs rule). Two things the original
plan didn't anticipate.

### 1. Package-mode smoke leg typechecks the worker IN ISOLATION, not the full consumer tree

The spec's Verification step 3 said the package-mode leg should confirm the
worker "typechecks". As implemented, the leg typechecks the **emitted
`src/worker.ts` alone** (against a minimal stub `AppModule` in a throwaway
tsconfig), not the whole consumer tree. Why the original "full-tree typecheck"
plan is impossible in package mode under checkout:

- In `runtime: package` mode the consumer's ENTIRE generated tree (repositories,
  services, `app.module.ts`) imports its base classes + tokens from
  `@pattern-stack/codegen/runtime/...`. In checkout mode that package is **not
  `bun add`-ed**, so every one of those specifiers is unresolvable â†’ the base
  classes collapse to `any` â†’ `extends IntegratedEntityRepository<â€¦>` breaks and
  cascades into `TS4112` (override-without-base) + `TS2339` (`baseQuery`/`create`/â€¦
  missing) across the whole tree. None of those are worker bugs; they are all
  downstream of the one unresolvable package import. (This is exactly why the
  existing subsystems smoke runs its full-tree typecheck only in the **vendored**
  leg, where everything is local.)
- So the package leg isolates: it copies the emitted `src/worker.ts` into a fresh
  dir beside a 3-line stub `AppModule`, points tsconfig `paths` at the consumer's
  `node_modules`, and runs `tsc --noEmit`. The worker's single package import
  (`@pattern-stack/codegen/runtime/subsystems/jobs/index`) is filtered with an
  **exact `'@pattern-stack/codegen/` substring match** â€” identical precedent to
  `run-smoke.ts`'s checkout-mode tolerance. The validator confirmed this filter
  scope is tight: it can only suppress the one unresolvable package-import line,
  so a real worker type error (in the relative `./app.module` import, the
  `forRoot(<opts>)` literal, the `WorkerAppModule` class, or the bootstrap body)
  still surfaces and fails the leg.
- **The genuine end-to-end gate** â€” typechecking the worker with the package
  ACTUALLY installed (so the `JobWorkerModule` import resolves for real) â€” belongs
  in the tarball/`SMOKE_TARBALL` post-publish harness, where the packed
  `@pattern-stack/codegen` is `npm install`-ed into the consumer. That extension
  is tracked as **issue #521** (out of scope here, matching the spec's "Out of
  scope" note on tarball coverage).

### 2. `main-hook` injection silently no-ops against `project init`'s `main.ts` (pre-existing â€” root cause is issue #520)

The package-mode leg deliberately does NOT assert the embedded-mode comment block
landed in the init-emitted `main.ts`, because that injection is **pre-existingly
broken in BOTH vendored and package mode** (it injects fine into a clean/minimal
`main.ts`, which the `subsystem.test.ts` unit case exercises). The original
in-PR diagnosis blamed an object-literal-after-anchor quirk in Hygen's `after:`
matcher â€” **that theory was wrong** (the validator empirically refuted it: a plain
`// note: true` comment also reproduces the no-op, and `{ flag: false }` injects
fine).

**Verified root cause (filed as issue #520):** the failure is a broken `skip_if`,
not the `after:` anchor. Two compounding defects:
1. `templates/subsystem/jobs/prompt.js` returns its context object WITHOUT a
   `mainHookInjected` key (it returns `workerExists`, `skipSchema`, etc. but omits
   this one), so `skip_if: "<%= mainHookInjected %>"` interpolates an undefined
   local and renders the **literal string `true`** regardless of the `''` the CLI
   passes on `--mainHookInjected` (captured empirically: `RENDERED_SKIP_IF=[true]`).
2. Hygen evaluates an inject-mode `skip_if` as a **regex against the whole target
   file**, so the rendered `true` matches any line containing the substring `true`
   â€” e.g. `project init`'s `main.ts` carries
   `swaggerOptions: { persistAuthorization: true }`, producing a false-positive
   skip.

This is a `main-hook.ejs.t`/`prompt.js` bug, NOT a #517 regression â€” #517 wires
the worker emission, which is unaffected. **The fix is owned by issue #520** (add
`mainHookInjected` to the `prompt.js` return so the `''`/`'true'` arg threads
through, mirroring `workerExists`); deferred from #517 since the guidance is
non-functional comment text and standalone `src/worker.ts` â€” the operationally
meaningful file â€” is always emitted.
