# Spec — #488: Sink emitter — typed find-side projection + null-coercion defaults (no casts)

## Goal

Replace the find-side (`findByExternalId`) one-line delegation in the generated
sink (`src/cli/shared/sink-emission-generator.ts`) with a generated
**projection → canonical reshaping view** that stays sound under the documented
canonical-widening seam, WITHOUT the loose `as` casts the hand-written swe-brain
sinks use (`message.sink.ts:58-74`, research Addendum item 3 find-side bullets).

Two concrete defects in the hand-written sinks that this issue addresses at the
generator level:

1. **json projection typing** — `message.sink.ts:66-69` casts
   `row.reactions as MessageReaction[] | null` because the projection types json
   columns as `unknown` (`prompt-extension.js TS_TYPE_MAP.json='unknown'`,
   `:209`/`:936-939`). The repo's no-cast rule applies to emitted code, so the
   generator must NOT emit an `as`. The honest resolution (given the `unknown`
   projection input — a hard constraint inherited from #487's binding fact (b))
   is **not** to invent a typed conversion; it is to keep json fields flowing
   through the canonical at the same `unknown` typing the projection gives them,
   and to **mark the typed-json narrowing as an author seam** when the author
   widens the canonical to a concrete json shape (`MessageReaction[]`).

2. **null-coercion defaults** — nullable projection columns map to non-null
   canonical fields on the find side (`authorExternalId ?? 'unknown'`,
   `text ?? ''`, `isAppAuthored ?? false`). Where the default is **derivable**
   from the column's TS type (a type-zero value: `'' / 0 / false`), the generator
   emits the `?? <default>`; where the non-null default is **genuine policy**
   (`'unknown'` is a sentinel, not a type-zero), it is a **clearly-marked seam** —
   the generator must NOT invent a value.

This issue **keeps the emit-once scaffold shape** (still one file, still
`SCAFFOLD_SENTINEL`-gated, find side + write side in the same `<entity>.sink.ts`).
The base/subclass split is **#491's** scope; whatever the find side emits here
must read sensibly as *scaffold-then-author-owned* today and survive the flip to
*regenerated base* later (so: explicit, deterministic, comment-gated seams — no
hidden state, no anchors).

## Approach

### The core tension (why the current one-liner is a trap)

Today the find side is:

```ts
async findByExternalId(userId, externalId): Promise<<E>Canonical | null> {
  const row = await this.repo.findByExternalIdProjected(externalId, this.provider);
  return row;
}
```

`findByExternalIdProjected` returns `<E>IntegrationProjection | null`
(`integrated-entity-repository.ts:155-170`). `return row` type-checks **only
because** `<E>Canonical` defaults to `<E>IntegrationProjection`
(`sink-emission-generator.ts:214`). The instant the author widens
`<E>Canonical` (the documented seam — e.g. to the surface package's
`CanonicalMessage` with typed `reactions: MessageReaction[]`, non-null
`authorExternalId: string`, and *no* local-only columns), `return row` becomes a
type error: `unknown` is not assignable to `MessageReaction[]`, `string | null`
is not assignable to `string`, and the projection carries columns the canonical
dropped.

So the find side must **reshape** projection → canonical by explicit field
enumeration — the same no-cast/no-spread discipline the write side uses (#487).

### What the generator emits — three field classes, deterministically

The generator already receives, per field, a **nullable-aware `tsType`** on
`SinkCopyThroughField` (`buildSinkInput` → `tsTypeFor`,
`adapter-emission-generator.ts:1151-1156`, `:1190-1193`). That `tsType` is the
only signal needed to classify a field. The reshaping view enumerates, in
deterministic order: `id`, `externalId`, copy-through fields, FK external keys
(if any — but see "Find side has NO FK keys" below), timestamps. For each:

| Field class (from `tsType`) | Emitted find-side line | Rationale |
|---|---|---|
| **non-null scalar** (`string`, `number`, `boolean`, `Date`) | `<f>: row.<f>,` | passthrough; projection is already non-null |
| **nullable scalar with derivable type-zero** (`string \| null` → `''`, `number \| null` → `0`, `boolean \| null` → `false`) | `<f>: row.<f> ?? <zero>,` | derivable coercion — the generator KNOWS the type-zero |
| **nullable scalar, non-type-zero default is policy** (e.g. `authorExternalId` wants `'unknown'`) | `<f>: row.<f>, /* SEAM: if canonical requires non-null, supply default */` *(see Seam policy)* | NOT invented — coercion only emitted for type-zero |
| **json** (`unknown` / `unknown \| null`) | `<f>: row.<f>,` | passes through at `unknown`; typed-narrowing is a marked seam |
| **local-only projection column not in canonical** | *(omitted from the view)* | dropped, like swe-brain drops `conversationExternalId` on the find side |

The load-bearing decision: **the generator emits `?? <default>` ONLY when the
default is the column type's zero value** (`string → ''`, `number → 0`,
`boolean → false`). It NEVER emits a non-type-zero default (`'unknown'`,
`'system'`, a sentinel id) because that is policy the generator cannot derive —
inventing one would silently corrupt the diff contract. Non-type-zero defaults
stay a marked author seam.

### Why the view stays SOUND today (the soundness invariant)

A subtlety: if the generator emits `text: row.text ?? ''` (coercing
`string | null → string`) but `<E>Canonical` *still defaults to the projection*
(which types `text` as `string | null`), the coerced view is still assignable
(`string` ⊆ `string | null`) — no error. And if the author widens the canonical
to `text: string`, the coercion is exactly what makes it assignable. **So the
type-zero coercions are safe in BOTH states** — projection-default canonical AND
widened canonical. This is the property that lets the find side ship as scaffold
today and remain correct after #491 regenerates it.

The json `unknown` passthrough is likewise sound against a projection-default
canonical (identity) and is the *only* no-cast option against a widened
`MessageReaction[]` canonical — where it does NOT type-check, that is the
intended compile-error signal that the author must fill the typed-json seam (see
"Why a compile error is the right outcome").

### Why a compile error (not a cast) is the right outcome for typed json

Given #487's binding fact (b) — json is `unknown` on the projection — there is
**no cast-free way** to turn `unknown` into `MessageReaction[]`. The generator
has two honest choices: (i) emit nothing typed and let json pass through as
`unknown`, producing a **compile error** the moment the author widens the
canonical to a concrete json type; or (ii) emit an `as` cast, violating the hard
no-cast rule. We choose (i). The compile error is a *feature*: it routes the
author to the one place a human judgment is genuinely required (the runtime shape
of a jsonb blob, which the type system cannot guarantee — see
`prompt-extension.js:194-197` on why json maps to `z.unknown()`). The emitted
header documents this explicitly so the error is self-explaining, and the seam
comment names the field. This is consistent with the stack's "drift is a compile
error, not a runtime mismatch" principle (research Conventions).

### Find side has NO FK external keys (asymmetry with the write side)

The write side (#487) maps FK *external* keys (`accountExternalId`) INTO the
write object because `<E>IntegrationWrite` carries them (`writeFkFields`). The
**projection does not** — `projectionFields` carries the *local FK column*
(`accountId`, `:940-943`), not the external key. So the find-side view enumerates
the **local FK columns** as ordinary scalars (they are `string` / `string | null`
and fall into the scalar rows above), NOT external keys. If the author's widened
canonical needs `accountExternalId` (the external key) on the find side, that is
a genuine seam — the local projection cannot supply it without a resolve, which
is policy. Mark it; do not invent. **This asymmetry is the single easiest thing
to get wrong** and must be called out in the emitted header.

### What stays the author seam (the irreducible 10%)

- **Typed-json narrowing** — `unknown → MessageReaction[]`. No cast-free
  generator solution exists; surfaced as a compile error + marked seam.
- **Non-type-zero null defaults** — `?? 'unknown'`, `?? 'system'`. Policy, not
  derivable. Marked seam (a labelled comment naming the field), never invented.
- **External-key reconstruction on the find side** — `accountExternalId` from a
  local `accountId` requires a resolve; policy. Marked seam.
- **Ownership enforcement** — the existing `// row.userId === userId` comment
  (`:225-227`) is unchanged; out of scope.

### Single well-typed conversion point

The reshaping is emitted **inline** in `findByExternalId` (one explicit object
literal mapping `row` → canonical), NOT as a separate helper function. Grounds:
(a) it keeps the emit-once file self-contained for the author to edit in place
today; (b) #491 will lift the whole method into the generated base unchanged — a
free-standing helper would just be extra surface to relocate; (c) the inline
object literal is the exact shape the write side already uses (`const write:
<E>IntegrationWrite = { … }`), so the find side mirrors it (`const view:
<E>Canonical = { … }; return view;`) for reviewer symmetry. The null-guard for
`row === null` short-circuits before the view is built.

## File-level plan

### Modify — `src/cli/shared/sink-emission-generator.ts`

Primary change, find-side only. Concretely:

- **Add a find-side view builder** alongside the existing write-body assembly
  (`:150-189`). It consumes the SAME `input.copyThroughFields` (with their
  nullable-aware `tsType`) the write side consumes, plus `id`/`externalId`/
  timestamps, and produces the reshaping lines per the field-class table above.
  Reuse the `USER_ID_FIELD` handling for consistency, but note the find side
  reads `row.userId` (it IS a projection column) — there is no param-shorthand
  asymmetry on the find side (that asymmetry is write-side only). The view's
  copy-through enumeration mirrors `projectionFields` ordering
  (`prompt-extension.js:933-950`: id, externalId, copy-through, local FK columns,
  timestamps) so it visibly tracks the projection shape.

  A small pure classifier on `tsType` drives the per-field line:
  - `endsWith(' | null')` AND base ∈ {`string`,`number`,`boolean`} →
    `<f>: row.<f> ?? <zero>,` (zero = `''`/`0`/`false`).
  - base is `unknown` (json, with/without `| null`) → `<f>: row.<f>,` (+ a
    one-line `// SEAM (typed json): …` label when nullable-or-not, see below).
  - otherwise (non-null scalar, `Date`, nullable non-type-zero like a custom
    sentinel — but the generator only knows type-zero, so a nullable string is
    type-zero `''`) → `<f>: row.<f>,`.

  Note: a nullable `string | null` is ALWAYS coerced to `''` here because `''`
  IS the type-zero for `string` — the generator does not know the author wants
  `'unknown'`. The header documents that `''` is the type-zero default and that
  policy sentinels (`'unknown'`) are an author override. This is the deliberate
  line between derivable and policy.

- **Rewrite the `findByExternalId` body (`:223-228`)** from `return row;` to:

  ```
  const row = await this.repo.findByExternalIdProjected(externalId, this.provider);
  if (row === null) return null;
  // The repo lookup is (provider, externalId)-scoped. If your external_id is not
  // globally unique, enforce ownership here (e.g. row.userId === userId).
  const view: <E>Canonical = {
    <reshaping lines>
  };
  return view;
  ```

  The existing ownership-enforcement comment is preserved verbatim. Local-only
  columns (projection columns absent from the canonical) cannot be known by the
  generator from `SinkEmitInput` alone — the generator enumerates the FULL
  projection shape, which IS assignable to the projection-default canonical
  today, and the author *removes* lines for dropped fields when they widen. The
  header documents this ("delete lines for canonical fields you do not carry").

- **Update the file header doc-comment (`:11-43`)** — living documentation
  (CLAUDE.md). Scope the edit to the **find-side wording only** (per the
  sequencing risk below — #487 owns the write-side/FK/json *write* wording). Add
  a "## Find-side reshaping (generated vs author seam)" block stating:
  - generated: explicit projection→canonical enumeration; type-zero null
    coercions (`'' / 0 / false`); json passes through at `unknown`.
  - author seam: typed-json narrowing (a compile error guides you); non-type-zero
    null defaults (`'unknown'` sentinels); find-side external-key reconstruction
    from a local FK column; dropping local-only columns when you widen the
    canonical.
  Keep the `SCAFFOLD_SENTINEL` framing intact (not deleted this issue).

- **Inline `findByExternalId` doc** — replace the current single comment with a
  short generated-vs-seam note pointing the author at the header. Whole-line `//`
  comments only (the `codeOnly` test helper strips whole-line `//`; a trailing
  inline comment would leak and could mask an ` as `).

### Possibly modify — `adapter-emission-generator.ts buildSinkInput` (VERIFY, likely no change)

The find side needs the SAME `copyThroughFields` (nullable-aware `tsType`) the
write side already receives — and it does (`:1151-1156`). The find side ALSO
needs `id`, `externalId`, and timestamps to enumerate the full projection shape;
`id`/`externalId` are constants the emitter can hard-code, but **timestamps**
(`createdAt`/`updatedAt`) are conditional on `hasTimestamps` in the template
(`:944-949`) and are NOT currently carried in `SinkEmitInput`.

**Decision for the implementer to confirm against the projection-default
canonical:** the projection-default canonical (`<E>Canonical =
<E>IntegrationProjection`) INCLUDES timestamps, so a find-side view that *omits*
them is still assignable (a subset is NOT assignable — TS requires all members).
Therefore the view MUST enumerate timestamps when the projection carries them, or
it fails to satisfy the projection-default canonical. Two options:
  1. **Carry a `hasTimestamps: boolean` (and local-FK-column list) on
     `SinkEmitInput`**, derived in `buildSinkInput` from `def` (it has the
     fields/relationships in scope). Cleanest; the find side enumerates exactly
     the projection columns. **Preferred** — the spec recommends this.
  2. Spread-free is required, so a `...row` shortcut is NOT allowed (no-spread
     invariant). Option 1 is effectively mandatory for soundness.

So `buildSinkInput` **likely DOES change**: add the projection-shape inputs
(`hasTimestamps`, the local FK column camelNames) to `SinkEmitInput` so the find
side can enumerate the full projection. The implementer must verify the exact set
against `projectionFields` (`prompt-extension.js:933-950`) — id, externalId,
copy-through, local FK columns, timestamps — and source them from the SAME
derivation (anti-drift; the local FK columns are `rel.camelField`, the
copy-through are already carried). If a clean enumeration of the projection shape
proves to need more than `copyThroughFields` exposes, extend `SinkEmitInput`
rather than re-deriving in the emitter.

## Interfaces

Likely one additive change to the emitter input (see `buildSinkInput` decision):

```typescript
export interface SinkEmitInput {
  // ... existing members unchanged ...
  /** Does the entity carry created_at/updated_at? Drives whether the find-side
   *  view enumerates timestamp columns (the projection includes them). */
  hasTimestamps?: boolean;            // NEW (default false)
  /** Local FK column camelNames (e.g. `accountId`) carried on the projection —
   *  enumerated as scalars in the find-side view. Sourced from the same
   *  `belongsTo` derivation as `projectionFields`. */
  localFkColumns?: SinkCopyThroughField[];   // NEW (default [])
}
```

No new exported types beyond optional `SinkEmitInput` members. The behavioral
delta is in the rendered string. Illustrative output for a message-shaped entity
(json `reactions`, nullable `text`/`authorExternalId`, declared `userId`,
timestamps), against the **projection-default canonical** (compiles today):

```typescript
async findByExternalId(userId: string, externalId: string): Promise<MessageCanonical | null> {
  const row = await this.repo.findByExternalIdProjected(externalId, this.provider);
  if (row === null) return null;
  // Reshape the local projection into the canonical the orchestrator diffs.
  // Generated: passthrough + type-zero null coercions (''/0/false) + json at `unknown`.
  // SEAM: typed-json narrowing, non-type-zero defaults, and external-key
  // reconstruction are author-owned — see the file header.
  const view: MessageCanonical = {
    id: row.id,
    externalId: row.externalId,
    text: row.text ?? '',                        // string | null → string (type-zero '')
    authorExternalId: row.authorExternalId ?? '', // string | null → string (type-zero ''; author may prefer 'unknown')
    isAppAuthored: row.isAppAuthored ?? false,   // boolean | null → boolean (type-zero false)
    reactions: row.reactions,                    // json → `unknown` (typed-json narrowing is a seam)
    channelId: row.channelId,                    // local FK column (scalar passthrough)
    userId: row.userId,
    createdAt: row.createdAt,
    updatedAt: row.updatedAt,
  };
  return view;
}
```

When the author widens `MessageCanonical` to the surface package's
`CanonicalMessage` (typed `reactions: MessageReaction[]`, non-null
`authorExternalId`, dropped `channelId` in favour of `channelExternalId`), TWO
lines become compile errors — `reactions` (`unknown` ⊄ `MessageReaction[]`) and a
missing `channelExternalId` — which is the intended seam signal. The
`authorExternalId ?? ''` line already satisfies a non-null canonical; the author
edits `''`→`'unknown'` only if they want the sentinel.

## Open questions

- **`hasTimestamps`/`localFkColumns` on `SinkEmitInput` — RESOLVED to "yes,
  add them" pending implementer confirmation.** The find-side view must satisfy
  the projection-default canonical, which includes timestamps + local FK columns;
  TS requires all members; spread is banned. So the emitter must enumerate the
  full projection shape, which means `SinkEmitInput` must carry it. The
  implementer confirms the exact set against `projectionFields`
  (`prompt-extension.js:933-950`) and sources it from the same derivation.
- **`string | null` always coerces to `''` — is that ever wrong?** It is the
  type-zero, so it is always *type-correct*; it is only *semantically* wrong when
  the author wants a sentinel (`'unknown'`). That is policy and the header flags
  it. We deliberately prefer a type-correct derivable default over leaving the
  field nullable-and-broken; the author overrides the literal in place
  (emit-once today; a base-method override after #491). If the team prefers
  "never coerce, always seam," that is a one-line change to the classifier — but
  it pushes every nullable field to a manual seam, defeating the issue's
  "emit derivable coercions" acceptance criterion. Spec recommends coerce-to-
  type-zero.
- **Marker style for seams** — a per-line `// SEAM:` label vs a single header
  block. Cosmetic; implementer's call. Whole-line `//` only (codeOnly safety).
  Tests assert the *content* exists somewhere, not the exact placement.

## Sequencing & risk

- **vs #487 (write-side mapping): #487 LANDS FIRST; this rebases onto it.** Both
  L0 issues edit `sink-emission-generator.ts`, and **both rewrite the file header
  doc-comment** — the highest-conflict region. Mitigation: this issue scopes its
  header edits to a **new "Find-side reshaping" block** and the find-side inline
  comment ONLY; it does NOT touch the write-side/FK/json *write* wording #487
  owns. The find-side body (`:223-228`) and the write-side body (`:150-189`,
  `:230-243`) are otherwise disjoint, so the code conflict is confined to the
  header banner — a clean rebase if each issue stays in its lane.
- **vs #491 (seam split): downstream.** The find-side method must lift into the
  generated base UNCHANGED. Keeping the reshaping inline (no free-standing helper)
  and comment-gated (no anchors/hidden state) is what makes that lift a no-op.
- **vs #490 (per-field exclusion knob): downstream.** Once #490 lets the
  generator KNOW a field is dropped, the find-side view can OMIT it automatically
  instead of the author deleting the line. Until then, dropping local-only
  columns on widening is a documented manual edit. No action here.
- **Risk — soundness against widened canonicals.** The type-zero coercions are
  assignable to BOTH the projection-default and a non-null-widened canonical, so
  they cannot regress. The json `unknown` passthrough is the deliberate
  compile-error seam. The one genuine hazard is FORGETTING a projection column
  (e.g. timestamps) so the view fails the projection-default canonical — covered
  by Test 5 and the `buildSinkInput` projection-shape inputs.
- **Risk — inventing a default.** Acceptance forbids invented values. The
  classifier emits `??` ONLY for type-zero defaults; json gets NO default; non-
  type-zero defaults are seams. Tests 2/4 lock this.
