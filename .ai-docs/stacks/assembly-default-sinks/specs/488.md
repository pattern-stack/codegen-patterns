# Spec â€” #488: Sink emitter â€” typed find-side projection + null-coercion defaults (no casts)

## Goal

Replace the find-side (`findByExternalId`) one-line delegation in the generated
sink (`src/cli/shared/sink-emission-generator.ts`) with a generated
**projection â†’ canonical reshaping view** that stays sound under the documented
canonical-widening seam, WITHOUT the loose `as` casts the hand-written swe-brain
sinks use (`message.sink.ts:58-74`, research Addendum item 3 find-side bullets).

Two concrete defects in the hand-written sinks that this issue addresses at the
generator level:

1. **json projection typing** â€” `message.sink.ts:66-69` casts
   `row.reactions as MessageReaction[] | null` because the projection types json
   columns as `unknown` (`prompt-extension.js TS_TYPE_MAP.json='unknown'`,
   `:209`/`:936-939`). The repo's no-cast rule applies to emitted code, so the
   generator must NOT emit an `as`. The honest resolution (given the `unknown`
   projection input â€” a hard constraint inherited from #487's binding fact (b))
   is **not** to invent a typed conversion; it is to keep json fields flowing
   through the canonical at the same `unknown` typing the projection gives them,
   and to **mark the typed-json narrowing as an author seam** when the author
   widens the canonical to a concrete json shape (`MessageReaction[]`).

2. **null-coercion defaults** â€” nullable projection columns map to non-null
   canonical fields on the find side (`authorExternalId ?? 'unknown'`,
   `text ?? ''`, `isAppAuthored ?? false`). **The generator emits NO coercion in
   the projection-default state.** A coercion is *only* needed when the canonical
   member is non-null, and in the projection-default state every member is exactly
   as nullable as its projection column â€” so a bare passthrough type-checks and is
   the correct, diff-sound output. Coercion becomes necessary (and TS *forces*
   the author to add it) only when they widen a member to non-null â€” which is the
   compile-error-on-widen seam. The generator therefore **never emits `??`** for
   nullable scalars; it passes them through bare. This matches the hand-written
   ground truth exactly (see "Diff-soundness" below) and is the Gate 1.5 blocker
   fix.

   > **Gate 1.5 blocker (resolved â€” Option (a)).** The original spec emitted a
   > blanket type-zero coercion (`string | null â†’ ?? ''`). That is *type*-sound
   > but **NOT diff-sound**: the orchestrator diffs find() output against the raw
   > adapter record (`execute-integration.use-case.ts:269-278`) via
   > `DeepEqualDiffer`, which **does not equate `null` and `''`**
   > (`deep-equal.differ.ts` â€” `isEqual` `:187-208`, `normalize` "preserves
   > zero-vs-null distinction" `:34-35`, empty-string guard `:220`). A blanket
   > coercion makes a legitimately-`null` adapter value diff `false` against a
   > local `''`, producing a **spurious upsert that never converges to noop**. See
   > Diff-soundness.

This issue **keeps the emit-once scaffold shape** (still one file, still
`SCAFFOLD_SENTINEL`-gated, find side + write side in the same `<entity>.sink.ts`).
The base/subclass split is **#491's** scope; whatever the find side emits here
must read sensibly as *scaffold-then-author-owned* today and survive the flip to
*regenerated base* later (so: explicit, deterministic, comment-gated seams â€” no
hidden state, no anchors).

## Approach

### The core tension (why the current one-liner is a trap)

Today the find side is:

```ts
async findByExternalId(userId, externalId): Promise<<E>Canonical | null> {
  const row = await this.repo.findByExternalIdProjected(externalId, this.provider);
  return row;
}
```

`findByExternalIdProjected` returns `<E>IntegrationProjection | null`
(`integrated-entity-repository.ts:155-170`). `return row` type-checks **only
because** `<E>Canonical` defaults to `<E>IntegrationProjection`
(`sink-emission-generator.ts:214`). The instant the author widens
`<E>Canonical` (the documented seam â€” e.g. to the surface package's
`CanonicalMessage` with typed `reactions: MessageReaction[]`, non-null
`authorExternalId: string`, and *no* local-only columns), `return row` becomes a
type error: `unknown` is not assignable to `MessageReaction[]`, `string | null`
is not assignable to `string`, and the projection carries columns the canonical
dropped.

So the find side must **reshape** projection â†’ canonical by explicit field
enumeration â€” the same no-cast/no-spread discipline the write side uses (#487).

### What the generator emits â€” BARE passthrough, deterministically

The generator already receives, per field, a **nullable-aware `tsType`** on
`SinkCopyThroughField` (`buildSinkInput` â†’ `tsTypeFor`,
`adapter-emission-generator.ts:1151-1156`, `:1190-1193`). The reshaping view
enumerates, in deterministic order matching `projectionFields`
(`prompt-extension.js:933-950`): `id`, `externalId`, copy-through fields, local FK
columns, timestamps. **Every field is a bare passthrough â€” `<f>: row.<f>,`** â€” in
the projection-default state:

| Field class (from `tsType`) | Emitted find-side line | Rationale |
|---|---|---|
| **non-null scalar** (`string`, `number`, `boolean`, `Date`) | `<f>: row.<f>,` | passthrough; projection is non-null, canonical member is non-null |
| **nullable scalar** (`string \| null`, `number \| null`, `boolean \| null`) | `<f>: row.<f>,` | **BARE â€” no `??`.** Projection member is `\| null`; the projection-default canonical member is also `\| null`; types match exactly. Coercion would only be needed for a non-null target, which does not exist yet â†’ adding one is both unnecessary AND diff-unsound (see below). |
| **json** (`unknown` / `unknown \| null`) | `<f>: row.<f>,` | passes through at `unknown`; typed-narrowing is a marked seam |
| **local-only projection column not in canonical** | *(enumerated today; author deletes on widen)* | dropped only when the author widens; see "Local-only columns" |

The load-bearing decision (Gate 1.5 blocker fix): **the generator emits NO `??`
at all in the projection-default state.** A coercion is *only* correct when the
target canonical member is non-null â€” a fact the generator cannot see while the
canonical defaults to the projection (every member is `| null` there). Emitting a
type-zero `?? ''` against a still-nullable target is gratuitous AND breaks the
differ (Diff-soundness below). The author adds `?? <default>` exactly when they
widen a member to non-null and TS forces the choice â€” that is the seam.

### Diff-soundness â€” why blanket type-zero coercion is WRONG (the blocker)

`findByExternalId`'s output is diffed against the **raw adapter record** by the
orchestrator:

```ts
// execute-integration.use-case.ts:269-278
const existing = await this.sink.findByExternalId(input.userId, change.externalId);
const diff = this.differ.diff(existing, change.record, change.providerChangedFields);
if (diff === 'noop') { /* short-circuit â€” no write */ }
```

`DeepEqualDiffer.isEqual` (`deep-equal.differ.ts:187-208`) does **not** equate
`null` and `''`: `null === ''` is false; `normalize` returns both unchanged
(`:210-213`) and explicitly "preserves zero-vs-null distinction" (`:34-35`); the
numeric-string path guards out empty strings (`:220`). So:

- Adapter legitimately yields `text: null`; the local row stores `null`.
- A blanket `text: row.text ?? ''` coerces find() output to `''`.
- `isEqual(null, '')` â†’ **false** â†’ the differ reports a spurious `{ from: '',
  to: null }` change â†’ spurious upsert writes the column â†’ next cycle diffs `''`
  (now local) vs `null` (adapter) again â†’ **never converges to noop.**

This is a silent infinite-rewrite bug on every nullable column, on every flat /
projection-default entity â€” the common case. The original spec never read the
differ; this revision does and cites it. **The fix is bare passthrough for
nullable members** (preserve `null` exactly), so `isEqual(null, null)` â†’ true â†’
correct noop.

### Ground truth confirms bare passthrough (your reference sink)

swe-brain `message.sink.ts:58-74` (the measured ground truth) coerces **only the
members its `CanonicalMessage` declares non-null**, and passes nullable members
through **bare**:

| swe-brain find-side member | canonical type | emitted |
|---|---|---|
| `authorExternalId` | `string` (non-null) | `row.authorExternalId ?? 'unknown'` (policy default) |
| `text` | `string` (non-null) | `row.text ?? ''` (type-zero default) |
| `isAppAuthored` | `boolean` (non-null) | `row.isAppAuthored ?? false` (type-zero default) |
| `authorEmail`, `subtype`, `editedAt`, `deletedAt`, `visibility`, `occurredAt`, `channelExternalId`, `threadExternalId` | `... \| null` (nullable) | **bare `row.<f>` â€” no `??`** |
| `reactions`, `files`, `mentionExternalIds` | `... \| null` (json, nullable) | `(row.<f> as â€¦[] \| null) ?? null` (cast = the json seam; `?? null` is a no-op preserving null) |

The original spec's blanket `string | null â†’ ?? ''` was **strictly more
aggressive than the reference** â€” it would have coerced the five+ bare-passthrough
members, breaking every one. The revised policy (bare passthrough for nullables;
coercion only where the canonical member is non-null, supplied by the author on
widen) matches the ground truth exactly. The json `?? null` no-op in the
reference is irrelevant to the generator (it preserves null either way); the
generator emits bare `row.<f>` for json and the typed narrowing (`as`, which we
replace with the compile-error seam) is the author's.

### Why the view stays SOUND today (the soundness invariant)

With bare passthrough, every member's type equals its projection column's type, so
the view is assignable to the projection-default canonical by construction (it IS
the projection shape, enumerated). When the author widens a member to non-null,
the bare line `<f>: row.<f>,` becomes a compile error (`T | null` âŠ„ `T`) â€” the
author adds `?? <default>` right there, choosing type-zero (`''`/`0`/`false`) or
a policy sentinel (`'unknown'`) per the member's semantics. **The generator never
chooses the default; the author does, at the exact member, guided by the compile
error.** This is the property that lets the find side ship as scaffold today and
remain correct (and diff-sound) after #491 regenerates it.

The json `unknown` passthrough is likewise sound against a projection-default
canonical (identity) and is the *only* no-cast option against a widened
`MessageReaction[]` canonical â€” where it does NOT type-check, that is the
intended compile-error signal that the author must fill the typed-json seam (see
"Why a compile error is the right outcome").

### Why a compile error (not a cast) is the right outcome for typed json

Given #487's binding fact (b) â€” json is `unknown` on the projection â€” there is
**no cast-free way** to turn `unknown` into `MessageReaction[]`. The generator
has two honest choices: (i) emit nothing typed and let json pass through as
`unknown`, producing a **compile error** the moment the author widens the
canonical to a concrete json type; or (ii) emit an `as` cast, violating the hard
no-cast rule. We choose (i). The compile error is a *feature*: it routes the
author to the one place a human judgment is genuinely required (the runtime shape
of a jsonb blob, which the type system cannot guarantee â€” see
`prompt-extension.js:194-197` on why json maps to `z.unknown()`). The emitted
header documents this explicitly so the error is self-explaining, and the seam
comment names the field. This is consistent with the stack's "drift is a compile
error, not a runtime mismatch" principle (research Conventions).

### Find side has NO FK external keys (asymmetry with the write side)

The write side (#487) maps FK *external* keys (`accountExternalId`) INTO the
write object because `<E>IntegrationWrite` carries them (`writeFkFields`). The
**projection does not** â€” `projectionFields` carries the *local FK column*
(`accountId`, `:940-943`), not the external key. So the find-side view enumerates
the **local FK columns** as ordinary scalars (they are `string` / `string | null`
and fall into the scalar rows above), NOT external keys. If the author's widened
canonical needs `accountExternalId` (the external key) on the find side, that is
a genuine seam â€” the local projection cannot supply it without a resolve, which
is policy. Mark it; do not invent. **This asymmetry is the single easiest thing
to get wrong** and must be called out in the emitted header.

### What stays the author seam (the irreducible 10%)

- **Typed-json narrowing** â€” `unknown â†’ MessageReaction[]`. No cast-free
  generator solution exists; surfaced as a compile error + marked seam.
- **ALL null-coercion defaults** â€” `?? ''`, `?? 0`, `?? false`, `?? 'unknown'`.
  The generator emits NONE of these. They are added by the author when (and only
  when) they widen a canonical member to non-null; TS forces the choice at the
  exact member via the compile error. (Revised per Gate 1.5: even the type-zero
  defaults are author-owned now â€” emitting them in the projection-default state is
  diff-unsound.)
- **External-key reconstruction on the find side** â€” `accountExternalId` from a
  local `accountId` requires a resolve; policy. Marked seam.
- **Ownership enforcement** â€” the existing `// row.userId === userId` comment
  (`:225-226`) is unchanged; out of scope.

### Single well-typed conversion point

The reshaping is emitted **inline** in `findByExternalId` (one explicit object
literal mapping `row` â†’ canonical), NOT as a separate helper function. Grounds:
(a) it keeps the emit-once file self-contained for the author to edit in place
today; (b) #491 will lift the whole method into the generated base unchanged â€” a
free-standing helper would just be extra surface to relocate; (c) the inline
object literal is the exact shape the write side already uses (`const write:
<E>IntegrationWrite = { â€¦ }`), so the find side mirrors it (`const view:
<E>Canonical = { â€¦ }; return view;`) for reviewer symmetry. The null-guard for
`row === null` short-circuits before the view is built.

## Interfaces

Likely one additive change to the emitter input (see `buildSinkInput` decision):

```typescript
export interface SinkEmitInput {
  // ... existing members unchanged ...
  /** Does the entity carry created_at/updated_at? Drives whether the find-side
   *  view enumerates timestamp columns (the projection includes them). */
  hasTimestamps?: boolean;            // NEW (default false)
  /** Local FK column camelNames (e.g. `accountId`) carried on the projection â€”
   *  enumerated as scalars in the find-side view. Sourced from the same
   *  `belongsTo` derivation as `projectionFields`. */
  localFkColumns?: SinkCopyThroughField[];   // NEW (default [])
}
```

No new exported types beyond optional `SinkEmitInput` members. The behavioral
delta is in the rendered string. Illustrative output for a message-shaped entity
(json `reactions`, nullable `text`/`authorExternalId`, declared `userId`,
timestamps), against the **projection-default canonical** (compiles today):

```typescript
async findByExternalId(userId: string, externalId: string): Promise<MessageCanonical | null> {
  const row = await this.repo.findByExternalIdProjected(externalId, this.provider);
  if (row === null) return null;
  // Reshape the local projection into the canonical the orchestrator diffs.
  // Generated: BARE passthrough (preserves null so the differ converges) + json
  // at `unknown`. SEAM (author-owned, surfaced as a compile error when you widen
  // the canonical): typed-json narrowing, non-null coercions (?? '' / 0 / false /
  // 'unknown'), and external-key reconstruction â€” see the file header.
  const view: MessageCanonical = {
    id: row.id,
    externalId: row.externalId,
    text: row.text,                  // string | null â€” bare; coerce on widen to non-null
    authorExternalId: row.authorExternalId,  // string | null â€” bare
    isAppAuthored: row.isAppAuthored,        // boolean | null â€” bare
    reactions: row.reactions,        // json â†’ `unknown`; typed-json narrowing is a seam
    channelId: row.channelId,        // local FK column (scalar passthrough)
    userId: row.userId,
    createdAt: row.createdAt,
    updatedAt: row.updatedAt,
  };
  return view;
}
```

This output is **diff-sound**: a `null` from the adapter for `text` diffs equal
against a local `null` (the differ does not equate `null`/`''`), so the cycle
converges to noop. When the author widens `MessageCanonical` to the surface
package's `CanonicalMessage` (typed `reactions: MessageReaction[]`, non-null
`text`/`authorExternalId`/`isAppAuthored`, dropped `channelId` in favour of
`channelExternalId`), the bare lines become compile errors â€” `reactions`
(`unknown` âŠ„ `MessageReaction[]`), `text` / `authorExternalId` / `isAppAuthored`
(`T | null` âŠ„ `T`), and a missing `channelExternalId`. Each error pins the exact
member where the author chooses the default (`?? ''`, `?? 'unknown'`, `?? false`)
or the resolve. **The generator never picks a default; the compile error routes
the human to the decision** â€” and only on widen, never spuriously in the flat
case.
