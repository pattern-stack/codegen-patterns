# Spec — #490: YAML schema knobs — sink delete policy + per-field exclusion (schema→parser→emitter)

## Goal

Thread the two common-policy knobs the swe-brain seam analysis justified
(research Addendum item 3) from **schema → parser → emitter**, so the two most
common author-subclass reasons become declarative YAML instead of hand-written TS:

1. **Delete policy** — `integration.sink.delete: soft | tombstone | noop`.
   `soft`/`tombstone` drive the repo's existing config-driven delete; `noop`
   emits the logged-no-op delete body (the swe-brain
   `MessageSink.softDeleteByExternalId` case, `message.sink.ts:114-124`) directly
   in the generated sink so no subclass override is needed for it.
2. **Per-field exclusion** — `integration.sink.exclude_fields:` declares fields
   that must NOT be vendor-sourced (swe-brain's segmentation-owned
   `conversationExternalId`, written `null` unconditionally; `message.sink.ts:95-98`,
   ADR-0008 §8). The field is dropped from the copy-through **on both
   derivations** (the sink write object AND the repo `writeColumns`/`writeFields`)
   and the declared sentinel/null is written instead — so re-ingest never
   clobbers a later-assigned value.

Strictly these two knobs. The arbitrary-code long tail stays the author-subclass
seam (#491 `sink-seam-split`); this issue does NOT add policy knobs for it.

This issue **keeps the emit-once scaffold shape** (still one file, still
`SCAFFOLD_SENTINEL`-gated). Whatever bodies this issue emits must read sensibly
as scaffold-then-author-owned today and survive #491's flip to a regenerated
`@generated` base unchanged — so the knob-driven bodies are deterministic and
comment-gated, no hidden state.

## Approach

### The load-bearing fact: two parallel derivations, both must agree

The integration write/projection surface is derived **twice** from the same raw
entity YAML, by two pieces of code that share no module:

| Derivation | Lives in | Produces |
|---|---|---|
| **Repo template** | `templates/entity/new/clean-lite-ps/prompt-extension.js` → `buildIntegrationSurface()` (`:870-975`) | `<Entity>IntegrationWrite` members (`writeFields`/`writeFkFields`), `<Entity>IntegrationProjection`, and the runtime `IntegrationUpsertConfig` literal (`writeColumns`, `softDelete`, …) emitted by `repository.ejs.t:50-118` |
| **Sink emitter** | `src/cli/shared/adapter-emission-generator.ts` → `buildSinkInput()` (`:1134-1175`) | `SinkEmitInput` (`copyThroughFields`, `fkExternalKeys`) consumed by `generateDefaultSink` (`sink-emission-generator.ts`) |

They are mirrors, not a shared function (one is a hygen `.js` template, the other
TS) — exactly the drift hazard the stack's anti-drift requirement targets and
that Gate 1.5 caught on #487 (the `fkWriteKey` divergence). **Both knobs in this
issue must be applied at BOTH derivations, or the two layers disagree** and the
disagreement is a runtime bug, not a compile error (see §Delete agreement and
§Exclusion clobber below for why each layer matters).

### Delete knob — semantics and the soft/tombstone/noop ↔ boolean reconciliation

The runtime config knob is a **boolean**, not tri-state:
`IntegrationUpsertConfig.softDelete` (`integration-upsert-config.ts:56-58`),
consumed by `softDeleteByExternalId` (`integrated-entity-repository.ts:179-199`):

- `softDelete: true`  → sets `deletedAt` (the "soft" semantics).
- `softDelete: false` → tombstone-by-clearing: nulls `external_id`/`provider`
  (the "tombstone" semantics).
- There is **no repo-level no-op** — `noop` is a *sink-level* concern (the sink
  short-circuits and never calls the repo delete).

So the tri-state YAML knob maps onto the existing two-state runtime knob plus a
sink-local third state:

| `integration.sink.delete` | repo `IntegrationUpsertConfig.softDelete` | sink `softDeleteByExternalId` body |
|---|---|---|
| `soft` *(default)* | `true` | delegates: `return this.repo.softDeleteByExternalId(externalId, this.provider);` (unchanged) |
| `tombstone` | `false` | delegates (unchanged body; the repo branch differs) |
| `noop` | *(unchanged — see Judgment call)* | logged no-op: `this.logger?.debug(...); return null;` — **no repo call** |

**The default.** Today `softDelete` is derived as
`!!hasSoftDelete = behaviorNames.includes('soft_delete')` (prompt-extension.js
`:917`, `:1109`). i.e. the repo's *current* default is `softDelete: false`
(tombstone) UNLESS the entity declares the `soft_delete` behavior. The new
`delete:` knob's default must preserve current behavior with **no knob
declared**: when `integration.sink.delete` is absent, the derivation is exactly
today's `!!hasSoftDelete`. The knob, when present, **overrides** that:
`delete: soft` → `softDelete: true`; `delete: tombstone` → `softDelete: false`;
`delete: noop` → leaves `softDelete` at its `!!hasSoftDelete` default (the repo
config value is irrelevant for noop because the sink never calls the repo delete
— but it is emitted unchanged to avoid a spurious config diff). This is the
**Zod default fence**: the schema MUST NOT default `delete` to a literal `soft`,
because that would silently flip every soft_delete-free entity from tombstone to
soft. Leave it `.optional()` (no `.default()`), and let the *emitter* resolve
absence to `!!hasSoftDelete`.

> **Judgment call (delete:noop and the repo config).** For `delete: noop` the
> sink never calls the repo, so `IntegrationUpsertConfig.softDelete` is dead for
> the delete path. Decision: **do NOT special-case `softDelete` for noop** —
> emit it at its `!!hasSoftDelete` default. Grounds: (a) the upsert path still
> uses the rest of the config; (b) forcing a value would invent a second
> meaning for a field whose semantics are "how the repo deletes", which noop
> bypasses; (c) keeping it default-derived means the noop knob touches ONLY the
> sink body, the smallest correct blast radius.

### Delete agreement — designing out the two-derivations bug (Gate 1.5)

The hazard: the sink body and the repo `softDelete` boolean are derived
separately; for `soft`/`tombstone` they MUST agree (the sink delegates to the
repo, so if the sink thinks "soft" but the repo config says `false`/tombstone,
the delete does the wrong thing silently). The mechanism that designs this out:

- **Single semantic mapping, mirrored verbatim at both derivations.** Define the
  mapping `delete → softDelete-boolean` once as a documented rule
  (`soft → true`, `tombstone → false`, absent → `!!hasSoftDelete`, `noop →
  !!hasSoftDelete`) and apply the IDENTICAL rule in `buildIntegrationSurface()`
  (prompt-extension.js, which emits the repo config) and — only where the sink
  needs it — in `buildSinkInput()`. The sink does NOT recompute `softDelete`; it
  carries the *resolved `delete` mode* enum and chooses its body from it. The
  repo config carries the *boolean*. The single shared truth is the YAML
  `delete` value + the documented mapping rule.
- **Contract test locks the agreement** (Tests §3): assert that for each of the
  three modes, the sink body and the repo config the SAME YAML would produce are
  consistent (`soft`→sink delegates + config `softDelete:true`; `tombstone`→sink
  delegates + config `softDelete:false`; `noop`→sink logged-no-op body + repo
  config `softDelete` = `!!hasSoftDelete`). This is the early warning; the
  delegation itself is the runtime net (a wrong boolean produces a wrong delete,
  observable in integration tests).

### Exclusion — end-to-end semantics, and why it MUST touch both derivations

swe-brain's ground truth (`message.sink.ts:95-98`): `conversationExternalId` is a
declared column, but the adapter never sets it; the sink writes
`conversationExternalId: null` **unconditionally**, and the hand-written comment
warns "re-ingest must not clobber" a later segmentation-assigned value.

The clobber hazard is concrete and is the reason the knob cannot be sink-local.
`integrationUpsertOne` writes **every `writeColumns` member on both insert AND
update** (`integrated-entity-repository.ts:101` `copyThrough[col] = w[col]`, then
`set: { ...copyThrough }` at `:114-117`). The no-clobber "only write when
non-null" protection at `:118-120` is **FK-resolver-only** — it does NOT apply to
plain copy-through columns. So:

- If exclusion were sink-local (only dropped from the sink write object) but the
  field stayed in `writeColumns`, the repo would still need the field on the
  write type, and a present-but-null value would clobber the column to null on
  **every** re-ingest — the exact bug swe-brain's manual subclass exists to
  avoid.

Therefore the exclusion knob's correct end-to-end behavior is:

1. **Drop the field from the repo derivation** — remove it from
   `buildIntegrationSurface().writeColumns` AND from `writeFields` (so the column
   is no longer copy-through-written by `integrationUpsertOne`, AND is no longer a
   `<Entity>IntegrationWrite` member). With the column out of `writeColumns`, the
   repo never writes it on upsert → a later segmentation-assigned value is
   preserved. **This is the actual no-clobber mechanism** (drop-from-writeColumns),
   not "write null carefully".
2. **Drop the field from the sink derivation** — remove it from
   `buildSinkInput().copyThroughFields` so the generated `write` object no longer
   enumerates `<field>: record.<field>`. (Because step 1 removed it from
   `<Entity>IntegrationWrite`, leaving it in the sink object would be a compile
   error — so step 2 is forced by step 1, which is the property that keeps the two
   in sync.)
3. **Optionally write the declared sentinel** — see the sentinel decision below.
   The field is excluded from copy-through regardless; whether the sink ALSO
   writes an explicit value depends on whether the field remains a (now optional)
   write member.

The field remains a **projection column** (it is a real DB column;
`projectionColumns`/`projectionFields` are NOT touched by exclusion). The find
side therefore still reads it from the row and passes it through bare — which
#488 already does (bare null-preserving passthrough). See §Find-side + diff
soundness for why this is correct.

#### Sentinel: `null` literal vs YAML-declared value — DECISION

The plan asks "declared null/sentinel (which?)". **Decision: support an optional
declared value, defaulting to `null`.** Schema shape (see File-level plan):

```yaml
integration:
  sink:
    exclude_fields:
      conversation_external_id: null        # write nothing → field stays out of the write surface entirely
      # OR a literal sentinel:
      some_field: "system"                  # field becomes an optional write member written with the literal
```

Two sentinel modes, both honest:

- **`null` (the common case, swe-brain's):** the field is fully removed from the
  write surface — dropped from `writeColumns`, `writeFields`, and the sink write
  object. The repo never writes it (no-clobber by construction); the DB default /
  prior value stands. The sink writes **nothing** for it. This is strictly better
  than swe-brain's `conversationExternalId: null` (which still sat in
  `writeColumns` and relied on the manual override) — the knob makes the
  no-clobber structural.
  > **Why not emit `<field>: null` in the sink for the `null` case?** Because the
  > field is removed from `<Entity>IntegrationWrite` (step 1), so an explicit
  > `conversationExternalId: null` would be a compile error (excess property).
  > "Excluded with null" means "absent from the write surface", which is the
  > correct no-clobber shape.
- **literal sentinel (`"system"`, `0`, `false`):** the field is removed from
  copy-through (`writeColumns`) BUT kept as an **optional** `<Entity>IntegrationWrite`
  member written with the declared literal by the sink (`<field>: 'system'`).
  This is rarer; it exists for "always write this constant, never the vendor
  value". Because the field is out of `writeColumns`, the repo would NOT write it
  via the generic copy-through path — so a literal sentinel requires the field to
  stay in the write surface AND be written. **This sub-case is the one place
  exclusion and the repo write path interact non-trivially; see Open question 1
  for the bounded scope decision.**

**Recommended scope for #490: ship the `null` mode as the primary, fully-specified
path** (it is swe-brain's measured need and the clean no-clobber story). Treat the
literal-sentinel mode as a thin extension only if it falls out cleanly; if it
requires touching the repo write path (it does — see OQ1), **defer the literal
sentinel to a follow-up** and ship `exclude_fields` accepting only `null` (or a
bare list) in #490. This keeps the issue proportionate.

> **Schema shape consequence of the scope decision:** if literal sentinels are
> deferred, `exclude_fields` is a **string array** (`exclude_fields:
> [conversation_external_id]`) — simplest, matches the `null`-only semantics, no
> ambiguity. If literal sentinels ship, it is a `Record<string, <literal>|null>`.
> **Recommend the string-array form for #490** and record the Record form as the
> follow-up's shape. The acceptance criterion "writes the declared null/sentinel"
> is satisfied by the string-array form because "excluded" ≡ "the declared null"
> ≡ "absent from the write surface" (the no-clobber outcome the criterion is
> really asking for).

### Find-side + diff-soundness (the constraint #488 imposes on excluded fields)

An excluded field is still a projection column, so the find side returns it. #488
makes the find side a **bare null-preserving passthrough** (`<f>: row.<f>,`, no
`??`, no cast) precisely so the differ sees `null` as `null`. For an excluded
field:

- The adapter never sends it (it is not on the canonical write surface).
- The local row holds whatever segmentation assigned (or the DB default / null).
- The differ (`execute-integration.use-case.ts:269-278` →
  `DeepEqualDiffer`) compares find() output against the adapter record's
  `providerChangedFields`. **An excluded field is NOT in the adapter's changed
  fields** (the adapter does not source it), so the differ never diffs it — there
  is no spurious-upsert risk for excluded fields. This is why dropping the field
  from the *write* surface while leaving it on the *projection* is diff-sound:
  the find side reports the true local value, and the differ only diffs fields the
  adapter actually changed. **#490 emits nothing special on the find side for
  excluded fields** — #488's bare passthrough already returns the local value,
  which is the correct, diff-sound answer. The spec must (and does) state this so
  a reviewer does not expect a find-side change here.

### Does exclusion thread into prompt-extension.js? — YES, and here is why (the plan's required verification)

The plan demands a verified answer to "does per-field exclusion thread into the
`writeFields`/`writeColumns` derivation in prompt-extension.js, or stay
sink-local?" **Answer: it MUST thread into prompt-extension.js**, for the
no-clobber reason proven above (a column left in `writeColumns` is clobbered on
every upsert; the FK-only no-clobber guard does not protect copy-through
columns). Leaving prompt-extension.js untouched is **not** a viable option for
the `null`/no-clobber case — it would ship the exact bug the knob exists to fix.

Concretely, `buildIntegrationSurface()` must filter the excluded fields out of
`processedFields` before computing `writeColumns`/`writeFields` (the projection
keeps them). The exclusion set must reach prompt-extension.js — it reads the raw
`definition` (`:992`), so `definition.entity.integration?.sink?.exclude_fields`
is in scope there with no new plumbing. The sink derivation (`buildSinkInput`)
reads the SAME `def.entity.integration.sink.exclude_fields` from
`EmitAdaptersEntity` and drops the same fields from `copyThroughFields`. One YAML
source, two mirrored filters — the same mirror-not-share pattern #487 used for
`fkWriteKey`, locked by a contract test (Tests §3).

## Interfaces

```typescript
// src/schema/entity-definition.schema.ts — attached to IntegrationConfigSchema
const SinkPolicySchema = z.object({
  delete: z.enum(['soft', 'tombstone', 'noop']).optional(), // no .default — see default fence
  exclude_fields: z.array(z.string()).optional(),
}).strict();
export type SinkPolicy = z.infer<typeof SinkPolicySchema>;
// IntegrationConfigSchema gains:  sink: SinkPolicySchema.optional()

// src/cli/shared/sink-emission-generator.ts
export interface SinkEmitInput {
  // …existing…
  /** Resolved delete behavior for the sink BODY. soft|tombstone|absent collapse
   *  to 'delegate' (both call the repo); 'noop' emits the logged-no-op body.
   *  The soft↔tombstone distinction is the repo config's concern, not the sink's. */
  deleteMode: 'delegate' | 'noop';
}

// shared mapping rule — one definition, mirrored across both derivations
// (prompt-extension.js for the repo boolean; the caller for the sink enum)
function resolveSoftDeleteBoolean(
  deleteKnob: 'soft' | 'tombstone' | 'noop' | undefined,
  hasSoftDelete: boolean,
): boolean {
  if (deleteKnob === 'soft') return true;
  if (deleteKnob === 'tombstone') return false;
  return hasSoftDelete; // absent OR noop → preserve current default
}
```

## Tests

`just test-unit` is the gate. Concretely:

1. **Schema validation** (`src/__tests__/schema/…` or the existing entity-schema
   test): `integration.sink.delete: noop|soft|tombstone` parses; an invalid value
   (`delete: hard`) is **rejected**; `exclude_fields: [conversation_external_id]`
   parses; `exclude_fields` naming a non-existent field is rejected by the
   `superRefine`.
2. **Sink emitter** (`sink-emission-generator.test.ts`): `deleteMode: 'noop'`
   emits the `return null;` no-op body (no repo delegation, no cast);
   `deleteMode: 'delegate'` emits the unchanged repo delegation. An excluded field
   does NOT appear in the generated `write` object. The no-cast/no-spread
   invariant still holds (existing `codeOnly` helper).
3. **Two-derivation contract test** (new, the anti-drift lock — the #487-style

## Risks & open questions

1. **Literal-sentinel mode (the `exclude_fields` value) — scope.** Writing a
   constant sentinel (`some_field: "system"`) requires the field to stay on the
   write surface AND be written by the repo — but exclusion removes it from
   `writeColumns`, so the generic copy-through path won't write it. Supporting it
   means either (a) a separate "always-write-literal" config list threaded into
   the repo write path, or (b) the sink writes it as an optional write member
   while the repo's `writeColumns` excludes it (but then the repo never persists
   it — broken). **Recommendation: ship `exclude_fields` as a string-array
   (`null`/no-clobber semantics only) in #490** — that is swe-brain's measured
   need and the clean story; defer the literal-sentinel variant to a follow-up
   with its own design (it touches the repo write path). The acceptance criterion
   "writes the declared null/sentinel" is met by the null/no-clobber form.
   **Resolve with Doug before impl if the literal form is wanted now.**
2. **`delete:noop` log line.** The generated sink has no logger
   (`(repo, provider)` ctor). The noop body is a silent `return null;` + comment.
   If an observable debug log is required, the sink needs a logger param, which
   widens the assembly binding (`assembly-emission-generator.ts:203-207`) — that
   is #491-adjacent (the binding line). **Recommendation: silent `return null;`
   for #490**; if a log is mandated, fold the logger-param into #491's seam work
   (where the binding is already in flux), not here.
3. **Sequencing vs #488 (impl in flight on `dug/488-sink-find-side-typing`).** This
   branch is created off `origin/main` which has #487 but NOT #488. #488 rewrites
   the find-side body; #490 does NOT touch the find side (§Find-side). The two
   edits are in the same file but different methods (find vs delete + the write
   `copyThroughFields` input is upstream of #490). Land order: #488 then #490, or
   rebase #490 onto #488 before impl — re-check the find-side block is untouched.
   Low conflict risk (disjoint methods), but the emitted header region is shared
   (both add header lines) — expect a header-block conflict, trivially resolved.
4. **Snake/camel normalization of `exclude_fields`.** YAML field names are
   snake_case; `copyThroughFields` is keyed by snake `name` in `buildSinkInput`
   (`:1152`) but emits `snakeToCamel(name)`. The exclude match must be done on the
   SAME key (snake `name`) in both derivations, or an exclusion silently no-ops.
   The contract test (Tests §3) must use a multi-word field
   (`conversation_external_id`) to catch a snake/camel mismatch — single-word
   fields would pass even with a normalization bug (the #487 lesson).
