# Spec — #491: Sink seam split — `@generated` base + emit-once author subclass; delete the sink sentinel; keep `INTEGRATION_SINK` valid; RFC-0002 §4 q2 revision note

## Goal

Flip the default sink from a single **emit-once scaffold** to a **two-file seam**:

1. **`<entity>.sink.generated.ts`** — `@generated` (regenerated via `writeIfChanged`),
   carrying `<Entity>SinkBase` with the full mechanical mapping (the upsert `write`
   object from #487, the find-side reshaping view from #488, knob-driven bodies
   from #490) **plus the narrow protected seams** an author overrides. A YAML
   field change now reflows into the base on every run — the central acceptance
   gain over emit-once.
2. **`<entity>.sink.ts`** — **emit-once** (`existsSync`-skip), the thin author
   subclass `class <Entity>Sink extends <Entity>SinkBase {}`. Regen never touches
   it; author overrides survive. Keeps the class name `<Entity>Sink` and the file
   path `<entity>.sink`, so the assembly binding
   (`new <Entity>Sink(repo, '<provider>')` from `../../sinks/<entity>.sink`,
   `assembly-emission-generator.ts:160-161,203-207`) is **unchanged**.

Delete the superseded sink scaffold path (no backwards compat, CLAUDE.md): the
sink emitter's `SCAFFOLD_SENTINEL` const + `// <CODEGEN-SCAFFOLD-V1>` banner
(`sink-emission-generator.ts:48`, emitted at the file head) and the
`existsSync(sinkPath) → scaffoldsSkipped` skip-on-the-mapping-file branch
(`adapter-emission-generator.ts:1064-1065`). The **adapter** scaffold sentinel
(`adapter-emission-generator.ts:203`, still used by the adapter scaffold at
`:562` / `:966`) is a separate concern — **leave it**.

Living docs in the same PR (CLAUDE.md): RFC-0002 §4 q2 dated revision note
re-opening "emit-once vs subclass" with the changed premise (the mapping is now
regenerated, so the base/subclass split is the ONLY way to regenerate mapping
while preserving overrides — no longer "a layer for uniformity"); and the
integration skill (`SKILL.md`, `consumer-patterns.md`) updated off the
"emit-once default sink scaffold" framing to the `@generated` base + emit-once
subclass model.

## The central design tension (resolved — read this first)

The merged single-file sink (`sink-emission-generator.ts` on `origin/main`)
carries **four author seams** woven into otherwise-mechanical method bodies:

| # | Seam | Where it lives today (single file) | What "regenerate the file" would clobber |
|---|---|---|---|
| 1 | **Canonical widening** | `export type <E>Canonical = <E>IntegrationProjection;` (the alias the author re-points to `CanonicalMessage`) | the author's widened alias |
| 2 | **FK activation** | the `<writeKey>: null,` + SEAM lines in the upsert `write` object (#487) — author replaces `null` with `record.<writeKey>` post-widen | the author's `record.<writeKey>` edits |
| 3 | **Typed-json narrowing** | the `// SEAM (typed json …)` find-view lines (#488) — author narrows `unknown → MyType[]` | the author's narrowing |
| 4 | **Null-coercion on widen** | bare find-view passthrough lines (#488) — author adds `?? ''` / `?? 'unknown'` where they widened a member to non-null | the author's coercions |

If `<entity>.sink.generated.ts` becomes the **regenerated** home of the upsert +
find bodies, then the **naive split fails**: a subclass that can only override
*whole methods* (`class <E>Sink extends <E>SinkBase { async upsertByExternalId(...) {…} }`)
forces an author who wants to widen the canonical and activate one FK line to
**copy the ENTIRE mechanical `write` object** (every scalar + json copy-through)
into author-owned code — exactly the mechanical mapping #491 exists to keep
regenerated. The whole point evaporates.

**The RFC-0003 precedent (`incremental-read.ts:179-288`) is the answer.**
`IncrementalReadBase<T, F, M>` owns ALL the machinery (streaming, filter,
hydrate, cursor) and the subclass fills **three `protected abstract` narrow
seams** (`enumerate`/`hydrate`/`toCanonical`, `:236-257`) plus overrides a few
**`protected` virtual hooks with safe defaults** (`matchesRef`/`matchesRecord`/
`filterFor`, `:262-276`) and **`protected readonly` config knobs**
(`hydrateConcurrency`, `:194`). The author never re-implements the streaming
loop. **This spec applies that exact shape to the sink.**

### The seam-surface design (what lives in base vs subclass; how each of the 4 seams survives regen)

**Base file `<entity>.sink.generated.ts` (`@generated`, `writeIfChanged`) — `<Entity>SinkBase`:**

- **Generic over the canonical type** so the subclass binds the widened type
  WITHOUT re-typing the methods. Mirrors `IncrementalReadBase<T,…>`:

  ```ts
  export abstract class <Entity>SinkBase<TCanonical extends <Entity>IntegrationProjection = <Entity>IntegrationProjection>
    implements IIntegrationSink<TCanonical> {
    constructor(
      protected readonly repo: <Entity>Repository,
      protected readonly provider: string,
    ) {}
    // ... three IIntegrationSink methods, composed from protected seams ...
  }
  ```

  The default type argument `= <Entity>IntegrationProjection` keeps the base
  self-satisfying (it compiles standalone, the smoke gate's whole-tree `tsc`
  passes on the base alone) and the projection-default subclass
  (`class <E>Sink extends <E>SinkBase {}`) inherits the projection-typed sink
  with **zero** author edits — the FK-free / flat case is a literal empty
  subclass body. **Seam #1 (canonical widening)** survives regen because the
  author binds the widened type at the SUBCLASS:
  `class MessageSink extends MessageSinkBase<CanonicalMessage> {}` — the alias
  lives in the author-owned (emit-once) file, never regenerated. The base never
  hardcodes a non-generic `<E>Canonical` alias; the alias seam *moves* to the
  subclass.

  > **Constraint check (`TCanonical extends <E>IntegrationProjection`).** The
  > bound must be the projection so that the base's mechanical bodies — which
  > read `record.<scalar>` (write) and assemble `view: TCanonical = { id, …
  > projection columns }` (find) — type-check against any widened `TCanonical`.
  > This holds for the measured ground truth: swe-brain's `CanonicalMessage`
  > carries every projection column the find view enumerates PLUS extras
  > (`reactions: MessageReaction[]`, non-null `text`). **But it does NOT hold
  > unconditionally** — a widened canonical that *drops* a projection column
  > (e.g. `CanonicalMessage` drops the local `channelId` in favour of
  > `channelExternalId`) is NOT a structural supertype of the projection, so
  > `extends <E>IntegrationProjection` would reject it. See Open question 1 for
  > the resolution (the bound is `Partial`-relaxed OR the find view is a
  > `protected` overridable hook — both keep seams #3/#4 narrow; the spec
  > recommends the hook).

- **The three `IIntegrationSink` methods are CONCRETE on the base**, composed
  from small `protected` seams so the subclass overrides a NARROW hook, never
  the whole method:

  - `upsertByExternalId` — concrete; provider assert + builds `write` via a
    `protected buildWrite(record): <E>IntegrationWrite` seam + calls
    `this.repo.integrationUpsertOne(write, this.provider)`. The mechanical
    scalar/json copy-throughs and the FK `<writeKey>: null` lines live INSIDE
    `buildWrite` on the base. **Seam #2 (FK activation)** survives regen as a
    `protected buildWrite` override on the subclass *only when the author needs
    it* — and even then they override `buildWrite`, not `upsertByExternalId`
    (the provider assert + repo call stay generated). The base `buildWrite`
    returns the all-null-FK write; the subclass `buildWrite` (rare) calls
    `super`-equivalent shape + sets the activated FK keys. **Most authors never
    touch it** (FK-free entities: empty subclass).

  - `findByExternalId` — concrete; null-guard + repo lookup + builds the
    canonical view via a `protected toCanonicalView(row): TCanonical` seam.
    The bare null-preserving passthrough (#488) lives in `toCanonicalView` on
    the base. **Seams #3 (typed-json narrowing) and #4 (null-coercion)**
    survive regen as a `protected toCanonicalView` override on the subclass —
    again, ONLY when the author widens `TCanonical`; the projection-default
    subclass inherits the bare view unchanged.

  - `softDeleteByExternalId` — concrete; body chosen by the **#490 delete
    knob** baked into the BASE (`'delegate'` → `repo.softDeleteByExternalId`;
    `'noop'` → logged-comment `return null`). Knob-driven bodies belong in the
    base (regenerated, declarative). The author overrides this whole method on
    the subclass only for the arbitrary-policy long tail (swe-brain's logger
    line) — the knob covers the common no-op, so the subclass override becomes
    rare.

**Subclass file `<entity>.sink.ts` (emit-once, `existsSync`-skip) — `<Entity>Sink`:**

```ts
import { <Entity>SinkBase } from './<entity>.sink.generated';

// Emit-once. Override protected seams to widen the canonical / activate FK keys
// / narrow typed json / set non-null coercions. The mechanical mapping lives in
// the @generated base and reflows on every codegen run.
export class <Entity>Sink extends <Entity>SinkBase {}
```

- For the flat/FK-free case (every `integration-patterns` fixture today) this is
  a **literal empty body** that compiles unedited for every entity shape — the
  acceptance lesson ("emit-once subclass must compile UNEDITED for every entity
  shape, FK-bearing included"). FK-bearing entities also compile empty: the base
  emits `<writeKey>: null` (write-safe; repo skips null FKs), so an un-widened
  FK subclass is correct, just not yet FK-active — identical to the merged
  single-file behavior, only now the un-edited part is regenerated.
- Keeps `@Injectable()` on the **base** (NestJS reads the decorator metadata
  from the concrete class actually provided; verify — see Open question 2). The
  subclass is what the assembly news up.

**Why this satisfies both halves of the tension:** the mapping is fully
`@generated` (drift-safe; YAML reflows) AND every one of the four seams survives
regen because each is a *narrow protected hook on the base* that the author
overrides *in the emit-once subclass* — never a whole-method copy. This is the
RFC-0003 "base owns machinery, author fills narrow seams" shape, not the
"abstract base + whole-method override" shape RFC-0002 §4 q2 rejected (and that
rejection was made when the whole sink was a scaffold; the premise has changed —
see the RFC revision note).

## Approach

### What moves where (mechanical lift from the merged single file)

The merged `generateDefaultSink` (`sink-emission-generator.ts`) already builds
the upsert `write` body, the find view, and the three methods as strings. #491
**re-partitions the SAME rendered strings across two `generate*` functions** —
it does NOT re-derive the mapping. Concretely:

| Merged single-file region (origin/main) | Lands in |
|---|---|
| header doc-comment (`:1-90`) | split: `@generated` banner + base doc → base file; short author-note → subclass file |
| `SCAFFOLD_SENTINEL` const (`:48`) + emitted banner | **DELETED** (no backwards compat) |
| `USER_ID_FIELD`, `sinkNames`, `SinkEmitInput` shapes | shared (unchanged) |
| `write` body assembly (`copyThroughLines`/`fkLines`/`writeBody`, `:~180-210`) | base `buildWrite` body |
| find view assembly (`findViewLines`/`findViewBody`, `:~215-260`) | base `toCanonicalView` body |
| `export type <E>Canonical = <E>IntegrationProjection;` | **moves to the subclass type parameter default** (base is generic) — the alias-as-seam becomes the type-arg-as-seam |
| `@Injectable() class <E>Sink implements IIntegrationSink` + 3 methods | base `abstract class <E>SinkBase<TCanonical = …> implements IIntegrationSink<TCanonical>` with concrete methods composed from `protected` seams |
| (new) | subclass `class <E>Sink extends <E>SinkBase {}` |

`buildSinkInput` (`adapter-emission-generator.ts:1175-1262`) is **unchanged** —
the base consumes the identical `SinkEmitInput` (copyThroughFields,
fkExternalKeys, localFkColumns, hasTimestamps). #490 (in flight) adds
`deleteMode` to that input; #491's base reads `deleteMode` to pick the
`softDeleteByExternalId` body. **#491 assumes #490's landed shape** — see
Sequencing.

### Emission-decision change (the regen-semantics flip)

In the sink block of `emitAdapters` (`adapter-emission-generator.ts:1062-1071`):

- **Base** `<entity>.sink.generated.ts` → `writeIfChanged` (regenerate every
  run; a YAML field change reflows the mapping). Goes onto
  `result.written` (the `@generated` bucket), NOT `scaffoldsWritten`.
- **Subclass** `<entity>.sink.ts` → `existsSync`-skip → `writeFile` if absent.
  Keeps the emit-once role; goes onto `result.scaffoldsWritten` /
  `scaffoldsSkipped` (the bucket the assembly-contract test inspects).
- **Delete** the current `if (existsSync(sinkPath)) result.scaffoldsSkipped.push(...)`
  guard *for the mapping file* — the mapping file is now always regenerated.
  The `existsSync`-skip moves to the subclass file only.

### Sentinel deletion scope (precise)

- **DELETE** in `sink-emission-generator.ts`: the `SCAFFOLD_SENTINEL` const
  (`:48`) and its emission as the first line of the sink text. The base carries
  a normal `@generated` banner (reuse `generatedBanner(sourceDesc)` from
  `adapter-emission-generator.ts:206` or an equivalent — see Open question 3 on
  whether the emitter imports it or the base gets its own banner string). The
  subclass carries a short "emit-once, author-owned" comment but **no
  `<CODEGEN-SCAFFOLD-V1>` sentinel** — its emit-once-ness is enforced by
  `existsSync` in the orchestrator, not by a sentinel scan (nothing scans the
  sentinel; the orchestrator already keys off file existence).
- **KEEP** in `adapter-emission-generator.ts`: the `SCAFFOLD_SENTINEL` const
  (`:203`) — it is still emitted by the **adapter** scaffold (`:562`) and
  skip-checked for the adapter (`:966`). Untouched.

### Consumer-facing collision semantics (stated honestly — no backwards compat, but be precise)

`entity new --all [--force]` against a consumer that already has hand-authored
`<entity>.sink.ts` files (swe-brain: **8** of them — `message`, `channel`,
`reaction`, `meeting`, `email`, `transcript`, `document`, `folder`, each a
standalone `export class <E>Sink implements IIntegrationSink<...>` carrying the
old `// <CODEGEN-SCAFFOLD-V1>` line):

1. **`<entity>.sink.ts` (subclass) emit — `existsSync` true → SKIPPED.** The
   consumer's hand-written standalone class survives **intact**. The assembly
   binding `new <Entity>Sink(repo, '<provider>')` still resolves (the class name
   is unchanged). **No breakage, no behavior change.** Verified against
   swe-brain's `message-integration.module.ts:14,38`
   (`import { MessageSink } from '../../sinks/message.sink'` /
   `new MessageSink(repo, 'slack')`).
2. **`<entity>.sink.generated.ts` (base) — NEW → written.** It exports an
   `<Entity>SinkBase` that the consumer's hand-written class does **not** extend.
   It is **dead but compiling**: the base imports only the repo + projection +
   write types + `IIntegrationSink` (no dependency on the consumer's class), so
   the whole-tree `tsc` smoke stays green; it is an orphan export, not an error.
   The consumer may ignore it indefinitely or migrate by changing their class to
   `extends <Entity>SinkBase<TheirCanonical>` and deleting the mechanical body
   they no longer need.

**Honest consequence:** a regen drops a new orphan `*.sink.generated.ts` next to
each hand-written sink. It does not delete, rename, or break the hand-written
sink, and it does not change runtime behavior. This is the no-backwards-compat
outcome stated truthfully — the old single-file sinks keep working; the new base
is opt-in. (Migrating swe-brain's 8 sinks to the subclass shape is a separate
consumer task, not in #491.)

### Fixture migration (this repo)

The `integration-patterns` fixtures emit `<entity>.sink.ts` today. After #491,
a fresh emit produces `<entity>.sink.generated.ts` (base) + `<entity>.sink.ts`
(empty subclass). The checked-in snapshot fixture
(`test/integration-emit/__snapshots__/snapshot.test.ts.snap`) and any
emit-into-tree harness must be re-baselined to the two-file shape. The
`assembly-contract.test.ts` structural assertions (currently asserting the
single-file sentinel + full body at `crm/sinks/contact.sink.ts`) split into
base-file assertions (mapping body, `@generated`, `extends`-able) + subclass
assertions (`class ContactSink extends ContactSinkBase`, emit-once). See Tests.

## Interfaces

No change to `SinkEmitInput` from #491 (it inherits #490's `deleteMode`
addition). The emitter's exported function surface changes:

```typescript
// src/cli/shared/sink-emission-generator.ts

/** Emit the @generated <Entity>SinkBase mapping file (<entity>.sink.generated.ts).
 *  Regenerated every run via writeIfChanged — a YAML field change reflows here. */
export function generateSinkBase(input: SinkEmitInput): string;

/** Emit the emit-once author subclass (<entity>.sink.ts):
 *  `export class <Entity>Sink extends <Entity>SinkBase {}`. existsSync-skipped. */
export function generateSinkSubclass(input: SinkEmitInput): string;
```

The emitted base class shape (the load-bearing interface):

```typescript
// <entity>.sink.generated.ts  (@generated — DO NOT EDIT)
@Injectable()
export abstract class <Entity>SinkBase<
  TCanonical extends <Entity>IntegrationProjection = <Entity>IntegrationProjection,
> implements IIntegrationSink<TCanonical> {
  constructor(
    protected readonly repo: <Entity>Repository,
    protected readonly provider: string,
  ) {}

  async findByExternalId(userId: string, externalId: string): Promise<TCanonical | null> {
    const row = await this.repo.findByExternalIdProjected(externalId, this.provider);
    if (row === null) return null;
    return this.toCanonicalView(row);
  }

  async upsertByExternalId(userId: string, record: TCanonical, provider: string): Promise<{ id: string; saved: TCanonical }> {
    if (provider !== this.provider) throw new Error(/* provider mismatch */);
    const write = this.buildWrite(record);
    const proj = await this.repo.integrationUpsertOne(write, this.provider);
    return { id: proj.id, saved: record };
  }

  async softDeleteByExternalId(_userId: string, externalId: string): Promise<{ id: string } | null> {
    // delete:delegate  → return this.repo.softDeleteByExternalId(externalId, this.provider);
    // delete:noop      → /* tombstone-preserving no-op */ return null;
  }
```

## Open questions

1. **`TCanonical extends <E>IntegrationProjection` bound vs `toCanonicalView`
   hook (the soundness question).** A widened canonical that *drops* a projection
   column (swe-brain's `CanonicalMessage` drops local `channelId` for
   `channelExternalId`) is NOT a structural supertype of the projection, so a
   strict `extends <E>IntegrationProjection` bound rejects the very widening the
   seam exists for. **Two resolutions, both keep the seams narrow:**
   - (a) **Relax the bound** to `TCanonical = <E>IntegrationProjection` (default,
     no `extends` constraint) and make `buildWrite`/`toCanonicalView` the typed
     boundary — the base's default `toCanonicalView` returns the projection-typed
     view, and the subclass override supplies the widened view. The base's
     default body needs a controlled type assertion at the single
     `toCanonicalView` return (the projection IS the default `TCanonical`, so it
     is sound) — but that reintroduces an ` as ` in *generated* code, which the
     no-cast rule forbids.
   - (b) **(RECOMMENDED)** Keep `TCanonical` generic with the default
     `= <E>IntegrationProjection` and **make `toCanonicalView` / `buildWrite` the
     `protected` seam methods returning the concrete projection/write types on
     the base** (NOT `TCanonical`), with `findByExternalId`/`upsertByExternalId`
     bridging. The projection-default subclass needs no override (the
     concrete-typed seam IS the canonical); the widened subclass overrides the
     seam returning its widened type. This avoids both the cast AND the
     over-tight bound. **The implementer must prototype the exact generic/bound/
     return-type combination that (i) compiles the empty subclass for the
     projection-default case with NO cast, (ii) compiles a widened-canonical
     subclass override with NO cast, (iii) keeps `implements
     IIntegrationSink<TCanonical>` honest.** This is the single highest-risk
     mechanical detail; resolve it with a throwaway `tsc` prototype before
     writing the emitter. Recommendation: (b).
2. **`@Injectable()` placement — base, subclass, or both?** NestJS resolves the
   provider it is told to construct (the assembly news up the subclass directly
   via `useFactory`, so DI metadata may not be required at all — the factory
   constructs it). Verify whether the `@Injectable()` decorator is needed on the
   base (for the inherited constructor metadata) or the subclass, or neither
   (because the binding is a `useFactory`, not class-token injection). The
   merged sink has `@Injectable()` on the single class; the safe default is to
   keep it on the base (decorator inheritance) AND confirm the smoke compiles.
   Low risk (the binding is a factory), but worth a one-line check.
3. **`@generated` banner source for the base.** Reuse
   `generatedBanner(sourceDesc)` (`adapter-emission-generator.ts:206`) — but it
   lives in the adapter emitter, not the sink emitter. Either export it for the
   sink emitter to import, or give the sink base its own one-line `@generated`
   banner string. Cosmetic; implementer's call. The base MUST carry an
   `@generated`/DO-NOT-EDIT banner (it is regenerated) and MUST NOT carry the
   deleted sentinel.
4. **`super`-composition for the FK-activation override.** When the author
   overrides `buildWrite` to activate one FK key, can they call a base helper to
   get the mechanical write and then set the FK key, or must they re-list the
   write? Cleanest: the base `buildWrite` is the mechanical write; the subclass
   override does `const w = super.buildWrite(record); return { ...w,
   channelExternalId: record.channelExternalId };` — BUT spread is banned in
   *generated* code; this override is *author* code (not generated), so spread is
   the author's call. The generated base never spreads. Document in the seam
   comment that the override is author-owned (spread allowed there). No generated
   spread.
5. **Does `result` need a new bucket for the base file?** The base is
   `@generated`; the natural bucket is `result.written` (alongside assembly
   modules/tokens). Confirm the assembly-contract test reads the base from
   `written` and the subclass from `scaffoldsWritten` — no new `EmitResult`
   field is strictly required, but a dedicated `sinkBasesWritten` could read
   cleaner. Implementer's call; keep `EmitResult` minimal unless the test needs
   the split.
