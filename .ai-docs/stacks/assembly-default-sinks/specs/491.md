# Spec — #491: Sink seam split — `@generated` base + emit-once author subclass; delete the sink sentinel; keep `INTEGRATION_SINK` valid; RFC-0002 §4 q2 revision note

## Goal

Flip the default sink from a single **emit-once scaffold** to a **two-file seam**:

1. **`<entity>.sink.generated.ts`** — `@generated` (regenerated via `writeIfChanged`),
   carrying two standalone default functions (`default<E>BuildWrite` / `default<E>ToCanonicalView`
   — the #487 write + #488 find mapping) plus `<Entity>SinkBase` (generic over
   `TCanonical`, three concrete methods incl. the #490 knob-driven delete, two
   abstract seams). A YAML field change now reflows into the base default functions
   on every run — the central acceptance gain over emit-once.
2. **`<entity>.sink.ts`** — **emit-once** (`existsSync`-skip), the author subclass
   `class <Entity>Sink extends <Entity>SinkBase { /* two one-line seam wirings */ }`.
   Regen never touches it; author overrides survive. Keeps the class name
   `<Entity>Sink` and the file path `<entity>.sink`, so the assembly binding
   (`new <Entity>Sink(repo, '<provider>')` from `../../sinks/<entity>.sink`,
   `assembly-emission-generator.ts:161,204-206`) is **unchanged**.

Delete the superseded sink scaffold path (no backwards compat, CLAUDE.md): the
sink emitter's `SCAFFOLD_SENTINEL` const + `// <CODEGEN-SCAFFOLD-V1>` banner
(`sink-emission-generator.ts:107`, emitted at the file head `:365`) and rewire the
`existsSync(sinkPath) → scaffoldsSkipped` branch (`adapter-emission-generator.ts:1073-1074`)
so it guards only the emit-once **subclass** file (the base is always regenerated).
The **adapter** scaffold sentinel (`adapter-emission-generator.ts:203`, still used
by the adapter scaffold at `:562` and skip-checked at `:975-976`) is a separate
concern — **leave it**.

Living docs in the same PR (CLAUDE.md): RFC-0002 §4 q2 dated revision note
re-opening "emit-once vs subclass" with the changed premise (the mapping is now
regenerated, so the base/subclass split is the ONLY way to regenerate mapping
while preserving overrides — no longer "a layer for uniformity"); and the
integration skill (`SKILL.md`, `consumer-patterns.md`) updated off the
"emit-once default sink scaffold" framing to the `@generated` base + emit-once
subclass model.

## The central design tension (resolved — read this first)

The merged single-file sink (`sink-emission-generator.ts` on main @ f9371e5)
carries **four author seams** woven into otherwise-mechanical method bodies:

| # | Seam | Where it lives today (single file) | What "regenerate the file" would clobber |
|---|---|---|---|
| 1 | **Canonical widening** | `export type <E>Canonical = <E>IntegrationProjection;` (the alias the author re-points to `CanonicalMessage`) | the author's widened alias |
| 2 | **FK activation** | the `<writeKey>: null,` + SEAM lines in the upsert `write` object (#487) — author replaces `null` with `record.<writeKey>` post-widen | the author's `record.<writeKey>` edits |
| 3 | **Typed-json narrowing** | the `// SEAM (typed json …)` find-view lines (#488) — author narrows `unknown → MyType[]` | the author's narrowing |
| 4 | **Null-coercion on widen** | bare find-view passthrough lines (#488) — author adds `?? ''` / `?? 'unknown'` where they widened a member to non-null | the author's coercions |

If `<entity>.sink.generated.ts` becomes the **regenerated** home of the upsert +
find bodies, then the **naive split fails**: a subclass that can only override
*whole methods* (`class <E>Sink extends <E>SinkBase { async upsertByExternalId(...) {…} }`)
forces an author who wants to widen the canonical and activate one FK line to
**copy the ENTIRE mechanical `write` object** (every scalar + json copy-through)
into author-owned code — exactly the mechanical mapping #491 exists to keep
regenerated. The whole point evaporates.

**The RFC-0003 precedent (`incremental-read.ts:179-288`) is the answer.**
`IncrementalReadBase<T, F, M>` owns ALL the machinery (streaming, filter,
hydrate, cursor) and the subclass fills **three `protected abstract` narrow
seams** (`enumerate`/`hydrate`/`toCanonical`, `:236-257`) plus overrides a few
**`protected` virtual hooks with safe defaults** (`matchesRef`/`matchesRecord`/
`filterFor`, `:262-276`) and **`protected readonly` config knobs**
(`hydrateConcurrency`, `:194`). The author never re-implements the streaming
loop. **This spec applies that exact shape to the sink.**

### The seam-surface design — PROVEN Shape C (Gate 1.5: tsc prototype settled the trilemma)

> **This is the empirically-verified shape. The Gate-1.5 reviewer built the
> deferred `tsc` prototype (TS 6.0.3, `--strict`, preserved at
> `/tmp/sink-trilemma/probe*.ts`) and proved:**
> - **Shape A FAILS** — a generic base class with a *default-bodied* seam that
>   builds a projection literal typed `TCanonical` is **TS2322**
>   (`probe.ts:64`): *"`{…projection…}` is assignable to the constraint of
>   `TCanonical`, but `TCanonical` could be instantiated with a different
>   subtype."* This was v1's primary §Interfaces sketch (it carried an inline
>   `as` placeholder — a no-cast-rule violation). **Deleted.**
> - **Shape C COMPILES cast-free** — abstract seams on the base (no default
>   body → nothing to type-check against `TCanonical`), the mechanical default
>   bodies emitted as **standalone typed functions** at concrete
>   `Projection`/`Write` types, and the emit-once subclass implementing each
>   seam as a **one-line wiring** to the default function. Both the
>   projection-default subclass AND a widened-canonical subclass compile with
>   **zero casts**.

**Base file `<entity>.sink.generated.ts` (`@generated`, `writeIfChanged`) — three parts:**

1. **Standalone default functions** at concrete projection/write types (the
   mechanical mapping lifted verbatim from the merged single-file emitter —
   #487 write body, #488 find view). They are the regenerated home of the
   mapping; a YAML field change reflows here. They take/return the *concrete*
   `<E>IntegrationProjection` / `<E>IntegrationWrite` — **no generic, no cast**:

   ```ts
   function default<E>ToCanonicalView(row: <E>IntegrationProjection): <E>IntegrationProjection {
     return { id: row.id, externalId: row.externalId, /* …copy-throughs (incl. excluded), local FK, timestamps… */ };
   }
   function default<E>BuildWrite(record: <E>IntegrationProjection): <E>IntegrationWrite {
     return { externalId: record.externalId, /* …scalar+json copy-throughs… */ /* <writeKey>: null + SEAM */ /* userId, */ };
   }
   ```

2. **`<Entity>SinkBase<TCanonical = <Entity>IntegrationProjection>`** — generic
   over the canonical, the three `IIntegrationSink` methods CONCRETE (provider
   assert, repo delegation, null-guard, `{id, saved}` shapes — the machinery),
   composed from **`protected abstract` seams typed at `TCanonical`**. The seams
   are abstract (declaration only) precisely so they may be typed `TCanonical`
   without a body to type-check — this is what dodges TS2322:

   ```ts
   export abstract class <Entity>SinkBase<TCanonical = <Entity>IntegrationProjection>
     implements IIntegrationSink<TCanonical> {
     constructor(protected readonly repo: <Entity>Repository, protected readonly provider: string) {}

     async findByExternalId(userId: string, externalId: string): Promise<TCanonical | null> {
       const row = await this.repo.findByExternalIdProjected(externalId, this.provider);
       if (row === null) return null;
       return this.toCanonicalView(row);
     }
     async upsertByExternalId(userId: string, record: TCanonical, provider: string): Promise<{ id: string; saved: TCanonical }> {
       if (provider !== this.provider) throw new Error(/* provider mismatch */);
       const write = this.buildWrite(record);
       const proj = await this.repo.integrationUpsertOne(write, this.provider);
       return { id: proj.id, saved: record };
     }
     async softDeleteByExternalId(_userId: string, externalId: string): Promise<{ id: string } | null> {
       // #490 deleteMode baked in: 'delegate' → return this.repo.softDeleteByExternalId(externalId, this.provider);
       //                          'noop'     → /* tombstone-preserving */ return null;
     }

     /** SEAM #2 (FK activation) — projection-default subclass wires default<E>BuildWrite. */
     protected abstract buildWrite(record: TCanonical): <Entity>IntegrationWrite;
     /** SEAM #3/#4 (typed-json narrowing / null-coercion on widen) — projection-default subclass wires default<E>ToCanonicalView. */
     protected abstract toCanonicalView(row: <Entity>IntegrationProjection): TCanonical;
   }
   ```

   Note `softDeleteByExternalId` is **concrete** (knob-driven, no seam — the
   delete body is fully declarative via #490's `deleteMode`); only `buildWrite`
   / `toCanonicalView` are abstract seams.

**Subclass file `<entity>.sink.ts` (emit-once, `existsSync`-skip) — `<Entity>Sink`:**

The subclass is **NOT a literal empty body** (v1's claim — corrected). It carries
the two generated **one-line seam wirings** to the default functions. It is still
emit-once, still compiles unedited for every entity shape (FK-bearing included),
and the mapping still reflows in the `.generated.ts` file (the wirings call the
regenerated functions):

```ts
import {
  <Entity>SinkBase,
  default<E>ToCanonicalView,
  default<E>BuildWrite,
} from './<entity>.sink.generated';

// Emit-once. The mechanical mapping lives in <entity>.sink.generated.ts and
// reflows on every codegen run. To WIDEN: change `extends <Entity>SinkBase` to
// `extends <Entity>SinkBase<YourCanonical>` and replace the seam body with your
// reshaping (activate FK keys / narrow typed json / coerce nulls). The default
// wirings below are the projection-default mapping.
export class <Entity>Sink extends <Entity>SinkBase {
  protected toCanonicalView(row: <Entity>IntegrationProjection): <Entity>IntegrationProjection {
    return default<E>ToCanonicalView(row);
  }
  protected buildWrite(record: <Entity>IntegrationProjection): <Entity>IntegrationWrite {
    return default<E>BuildWrite(record);
  }
}
```

**How each of the four seams survives regen (Shape C):**

| Seam | Survives regen because… |
|---|---|
| **#1 canonical widening** | the author binds the widened type at the SUBCLASS (`extends <E>SinkBase<CanonicalMessage>`) — the type-arg lives in the emit-once file, never regenerated. The base default `= <E>IntegrationProjection` makes the un-widened case need no change. |
| **#2 FK activation** | the author overrides `buildWrite` IN THE SUBCLASS (replaces the `default<E>BuildWrite(record)` wiring with their own body activating `<writeKey>: record.<writeKey>`). The mechanical default function in the base still regenerates; the subclass just stops calling it. |
| **#3 typed-json narrowing** | the author overrides `toCanonicalView` in the subclass (narrows `unknown → MyType[]`); the base default function regenerates untouched. |
| **#4 null-coercion on widen** | same `toCanonicalView` override; TS forces a coercion only when the author widened a member to non-null (the compile error pins the member). |

**Proven sketch (embedded as evidence — `probe_BConly.ts` Shape C, compiles clean
under TS 6.0.3 `--strict`, exit 0, NO cast):**

```ts
// --- in <entity>.sink.generated.ts (@generated) ---
function defaultToCanonicalView(row: MessageIntegrationProjection): MessageIntegrationProjection {
  return { id: row.id, externalId: row.externalId, text: row.text,
           channelId: row.channelId, createdAt: row.createdAt, updatedAt: row.updatedAt };
}
function defaultBuildWrite(record: MessageIntegrationProjection): MessageIntegrationWrite {
  return { externalId: record.externalId, text: record.text, channelExternalId: null };
}
abstract class C_Base<TCanonical = MessageIntegrationProjection>
  implements IIntegrationSink<TCanonical> {
  constructor(protected readonly repo: MessageRepository, protected readonly provider: string) {}
  async findByExternalId(_u: string, e: string): Promise<TCanonical | null> {
    const row = await this.repo.findByExternalIdProjected(e, this.provider);
    if (row === null) return null;
    return this.toCanonicalView(row);
  }
  async upsertByExternalId(_u: string, record: TCanonical, provider: string): Promise<{ id: string; saved: TCanonical }> {
    if (provider !== this.provider) throw new Error("x");
    const proj = await this.repo.integrationUpsertOne(this.buildWrite(record), this.provider);
    return { id: proj.id, saved: record };
  }
  async softDeleteByExternalId(_u: string, e: string): Promise<{ id: string } | null> {
    return this.repo.softDeleteByExternalId(e, this.provider);
  }
  protected abstract toCanonicalView(row: MessageIntegrationProjection): TCanonical;
  protected abstract buildWrite(record: TCanonical): MessageIntegrationWrite;
}
// --- PROJECTION-DEFAULT subclass (emit-once) — one-line wirings, NO edit ---
class MessageSink_Default extends C_Base {
  protected toCanonicalView(row: MessageIntegrationProjection): MessageIntegrationProjection { return defaultToCanonicalView(row); }
  protected buildWrite(record: MessageIntegrationProjection): MessageIntegrationWrite { return defaultBuildWrite(record); }
}
// --- WIDENED subclass (emit-once, author-edited) — seams reimplemented, NO cast ---
class MessageSink_Widened extends C_Base<CanonicalMessage> {
  protected toCanonicalView(row: MessageIntegrationProjection): CanonicalMessage {
    return { id: row.id, externalId: row.externalId, text: row.text ?? "",   // seam #4 null-coerce
             channelExternalId: null, reactions: [],                          // seam #2 / #3
             createdAt: row.createdAt, updatedAt: row.updatedAt };
  }
  protected buildWrite(record: CanonicalMessage): MessageIntegrationWrite {
    return { externalId: record.externalId, text: record.text, channelExternalId: record.channelExternalId };
  }
}
```

> **Why Shape C beats the v1 generic-default sketch (Shape A).** In Shape A the
> base default `toCanonicalView` body builds a projection literal but is *declared*
> to return `TCanonical` — TS2322, because a subclass could instantiate
> `TCanonical` as an unrelated subtype the projection literal does not satisfy.
> Shape C sidesteps this entirely: the **abstract** seam has no body (nothing to
> check), and the mechanical body lives in a **standalone function** typed at the
> *concrete* projection (where `{…}` literally IS the return type — trivially
> assignable, no cast). The generic-vs-concrete tension is resolved by moving the
> default body OUT of the generic class into a concrete function the subclass
> wires in. This is also why the bound is **`TCanonical = <E>IntegrationProjection`
> with NO `extends` constraint** — a `extends <E>IntegrationProjection` bound would
> reject a widened canonical that drops a projection column (e.g. `CanonicalMessage`
> drops local `channelId`); the unconstrained generic accepts any widened shape,
> and the subclass's own `toCanonicalView` override is where the (now explicit)
> reshaping happens. The v1 "Constraint check" worry and OQ1 are RESOLVED by the
> probe: no bound, abstract seams, concrete default functions.

**`@Injectable()` (NOTE 3 / OQ2 — CLOSED).** The assembly binds the sink via
`useFactory: (repo) => new <E>Sink(repo, '<provider>')`
(`assembly-emission-generator.ts:204-206`) — it **constructs the subclass
directly**; NestJS class-token DI never instantiates the sink, so the
`@Injectable()` decorator is **decorative** (no DI metadata is read). Decision:
**omit `@Injectable()` from the base** (it is dead under a `useFactory`), keeping
the generated base lean; if a future change ever binds the sink by class token,
re-add it then. (The merged single-file sink carries `@Injectable()` cosmetically;
dropping it is correct, not a regression — the factory binding is unchanged.)

**Why this satisfies both halves of the tension:** the mapping is fully
`@generated` (the default functions reflow on every run) AND every one of the four
seams survives regen because each is a *narrow seam* the author reimplements in
the emit-once subclass — never a whole-method copy of the machinery (the provider
assert, repo delegation, and return shapes always stay in the base). This is the
RFC-0003 "base owns machinery, author fills narrow seams" shape, not the
"abstract base + whole-method override" shape RFC-0002 §4 q2 rejected (and that
rejection was made when the whole sink was a scaffold; the premise has changed —
see the RFC revision note).

## Approach

### What moves where (mechanical lift from the merged single file)

The merged `generateDefaultSink` (`sink-emission-generator.ts`) already builds
the upsert `write` body, the find view, and the three methods as strings. #491
**re-partitions the SAME rendered strings across two `generate*` functions** —
it does NOT re-derive the mapping. Concretely:

All line numbers below are anchored to **main @ f9371e5** (post-#490).

| Merged single-file region (main @ f9371e5) | Lands in |
|---|---|
| header doc-comment (`sink-emission-generator.ts:1-90`) | split: `@generated` banner + base doc → base file; short author-note → subclass file |
| `SCAFFOLD_SENTINEL` const (`:107`) + its emission as the sink's first line (`:365`) | **DELETED** (no backwards compat) |
| `USER_ID_FIELD`, `sinkNames`, `SinkEmitInput` shapes | shared (unchanged) |
| `write` body assembly (`copyThroughLines`/`fkLines`/`writeBody`, `:~270-299`) | base `default<E>BuildWrite` standalone function |
| find view assembly (`viewFields`/`findViewLines`/`findViewBody`, `:322-345`) | base `default<E>ToCanonicalView` standalone function |
| `softDelete` `deleteBody` branch (`:351-364`, `deleteMode` `'delegate'`/`'noop'`) | base `softDeleteByExternalId` CONCRETE method (no seam — declarative knob) |
| `export type <E>Canonical = <E>IntegrationProjection;` (`:~388`) | **DELETED** — replaced by the base's generic `TCanonical = <E>IntegrationProjection` default + the subclass's own type-arg (seam #1 moves to the subclass) |
| `@Injectable() class <E>Sink implements IIntegrationSink<<E>Canonical>` + 3 methods | base `abstract class <E>SinkBase<TCanonical = <E>IntegrationProjection> implements IIntegrationSink<TCanonical>` — 3 CONCRETE methods + 2 `protected abstract` seams (no `@Injectable` — OQ2 CLOSED) |
| (new) | standalone `default<E>BuildWrite` / `default<E>ToCanonicalView` functions in the base file |
| (new) | subclass `class <E>Sink extends <E>SinkBase { /* 2 one-line seam wirings */ }` |

`buildSinkInput` (`adapter-emission-generator.ts:1184`, **exported** since #490)
is **unchanged** — the base consumes the identical landed `SinkEmitInput`:
`copyThroughFields` (write, post-exclusion), `viewCopyThroughFields` (find view,
FULL projection — #490 Gate 2.5), `fkExternalKeys`, `localFkColumns`,
`hasTimestamps`, `deleteMode?: 'delegate' | 'noop'`. The base's
`default<E>BuildWrite` consumes `copyThroughFields` + `fkExternalKeys`; the base's
`default<E>ToCanonicalView` consumes `viewCopyThroughFields ?? copyThroughFields`
(the landed fallback at `:322`) + `localFkColumns` + `hasTimestamps`; the base's
`softDeleteByExternalId` reads `deleteMode`. **No new `SinkEmitInput` member is
required by #491** — it re-partitions the rendered strings, nothing more.

> **#490 Gate 2.5 reconciliation (BLOCKER 2 premise change — SIMPLIFIES this
> design).** #490's Gate 2.5 correction (specs/490.md:733-770, landed PR #497)
> made exclusion **write-surface-only**: the find view **always enumerates the
> FULL projection** (excluded fields stay as bare passthroughs via
> `viewCopyThroughFields`); diff-soundness holds via the differ's `key in
> incoming` guard (`deep-equal.differ.ts:159`), not via dropping the field from
> the view. **Consequence for #491:** the lifted `default<E>ToCanonicalView` is
> **knob-independent** — it is the full-projection bare passthrough regardless of
> `exclude_fields`. The ONLY knob the base branches on is `deleteMode` (in the
> concrete `softDeleteByExternalId`). The write-side `buildWrite` consumes the
> already-post-exclusion `copyThroughFields` (the orchestrator's `buildSinkInput`
> did the filtering). #491 therefore inherits a clean, knob-orthogonal find view —
> no exclusion logic in the lifted view code at all.

### Emission-decision change (the regen-semantics flip)

In the sink block of `emitAdapters` (`adapter-emission-generator.ts:1071-1080`):

- **Base** `<entity>.sink.generated.ts` → `writeIfChanged` (regenerate every
  run; a YAML field change reflows the mapping). Goes onto
  `result.written` (the `@generated` bucket — alongside the assembly modules at
  `:999`/`:1015`), NOT `scaffoldsWritten`.
- **Subclass** `<entity>.sink.ts` → `existsSync`-skip → `writeFile` if absent.
  Keeps the emit-once role; goes onto `result.scaffoldsWritten` (`:1079`) /
  `scaffoldsSkipped` (`:1074`) — the bucket the assembly-contract test inspects.
- **The current** `if (existsSync(sinkPath)) result.scaffoldsSkipped.push(sinkPath)`
  guard (`:1073-1074`) is rewired: it now guards the **subclass** file only. The
  base file is always regenerated via `writeIfChanged` (no existsSync gate).

### Sentinel deletion scope (precise — anchored to main @ f9371e5)

- **DELETE** in `sink-emission-generator.ts`: the `SCAFFOLD_SENTINEL` const
  (`:107`) and its emission as the first line of the sink text (the
  `` return `${SCAFFOLD_SENTINEL}\n…` `` at `:365`). The base carries a normal
  `@generated` banner (see Open question 3 on whether the emitter imports
  `generatedBanner` from `adapter-emission-generator.ts:206` or the base gets its
  own banner string). The subclass carries a short "emit-once, author-owned"
  comment but **no `<CODEGEN-SCAFFOLD-V1>` sentinel** — its emit-once-ness is
  enforced by `existsSync` in the orchestrator, not by a sentinel scan (nothing
  scans the sentinel; the orchestrator already keys off file existence).
- **KEEP** in `adapter-emission-generator.ts`: the `SCAFFOLD_SENTINEL` const
  (`:203`) — it is still emitted by the **adapter** scaffold (`:562`) and
  skip-checked for the adapter (`existsSync(scaffoldPath)` at `:975`,
  `scaffoldsSkipped.push(scaffoldPath)` at `:976`). Untouched.

### Consumer-facing collision semantics (stated honestly — no backwards compat, but be precise)

`entity new --all [--force]` against a consumer that already has hand-authored
`<entity>.sink.ts` files (swe-brain: **8** of them — `message`, `channel`,
`reaction`, `meeting`, `email`, `transcript`, `document`, `folder`, each a
standalone `export class <E>Sink implements IIntegrationSink<...>` carrying the
old `// <CODEGEN-SCAFFOLD-V1>` line):

1. **`<entity>.sink.ts` (subclass) emit — `existsSync` true → SKIPPED.** The
   consumer's hand-written standalone class survives **intact**. The assembly
   binding `new <Entity>Sink(repo, '<provider>')` still resolves (the class name
   is unchanged). **No breakage, no behavior change.** Verified against
   swe-brain's `message-integration.module.ts:14,38`
   (`import { MessageSink } from '../../sinks/message.sink'` /
   `new MessageSink(repo, 'slack')`).
2. **`<entity>.sink.generated.ts` (base) — NEW → written.** It exports an
   `<Entity>SinkBase` + the two `default<E>*` functions that the consumer's
   hand-written class does **not** import. It is **dead but compiling**: the base
   imports only the repo + projection + write types + `IIntegrationSink` (no
   dependency on the consumer's class), so the whole-tree `tsc` smoke stays
   green; it is an orphan module, not an error. The consumer may ignore it
   indefinitely or migrate by changing their class to
   `extends <Entity>SinkBase<TheirCanonical>` and wiring/overriding the seams.

> **NOTE 2 (reviewer) — there is NO sinks barrel.** Sinks are imported by path
> (`assembly-emission-generator.ts:161` → `../../sinks/<entity>.sink`); no
> `sinks/index.ts` is emitted (confirmed: no `sinks/index` reference in
> `adapter-emission-generator.ts`). So the "orphan double-export" concern from v1
> is **moot** — there is no barrel that would re-export both files and collide.
> The orphan-base-compiles property is gated by `just test-smoke-integration` for
> the in-repo tree (whole-tree `tsc`); for consumer trees it is *reasoned* (the
> base is a self-contained module that imports nothing of the consumer's), not
> CI-gated here (swe-brain has its own typecheck).

**Honest consequence:** a regen drops a new orphan `*.sink.generated.ts` next to
each hand-written sink. It does not delete, rename, or break the hand-written
sink, and it does not change runtime behavior. This is the no-backwards-compat
outcome stated truthfully — the old single-file sinks keep working; the new base
is opt-in. (Migrating swe-brain's 8 sinks to the subclass shape is a separate
consumer task, not in #491.)

### Fixture migration (this repo)

```typescript
import type { IIntegrationSink } from '<subsystems import>';
import { <Entity>Repository, type <Entity>IntegrationProjection, type <Entity>IntegrationWrite } from '<repoImportSpecifier>';

// Mechanical defaults at CONCRETE projection/write types — the literal IS the
// return type, so NO cast is needed (this is what dodges the TS2322 of a
// generic default body). The regenerated home of the #487 write / #488 view.
export function default<E>ToCanonicalView(row: <Entity>IntegrationProjection): <Entity>IntegrationProjection {
  return {
    id: row.id,
    externalId: row.externalId,
    // ... bare passthrough: copy-through (full projection incl. excluded), local FK, timestamps ...
    // ... // SEAM (typed json — unknown; narrow on canonical widen): <field>  for json cols ...
  };
}
export function default<E>BuildWrite(record: <Entity>IntegrationProjection): <Entity>IntegrationWrite {
  return {
    externalId: record.externalId,
    // ... scalar + json copy-throughs (#487, post-exclusion) ...
    // ... <writeKey>: null,  + SEAM comment (#487 — null until canonical widens) ...
    // ... userId,  (when user_id declared) ...
  };
}

export abstract class <Entity>SinkBase<TCanonical = <Entity>IntegrationProjection>
  implements IIntegrationSink<TCanonical> {
  constructor(
    protected readonly repo: <Entity>Repository,
    protected readonly provider: string,
  ) {}

  async findByExternalId(userId: string, externalId: string): Promise<TCanonical | null> {
    const row = await this.repo.findByExternalIdProjected(externalId, this.provider);
    if (row === null) return null;
    return this.toCanonicalView(row);
  }

  async upsertByExternalId(userId: string, record: TCanonical, provider: string): Promise<{ id: string; saved: TCanonical }> {
    if (provider !== this.provider) {
      throw new Error(`<Entity>Sink: bound provider '${this.provider}' != run provider '${provider}'`);
    }
    const proj = await this.repo.integrationUpsertOne(this.buildWrite(record), this.provider);
    return { id: proj.id, saved: record };
  }

  async softDeleteByExternalId(_userId: string, externalId: string): Promise<{ id: string } | null> {
    // CONCRETE — #490 deleteMode (no seam):
    //   'delegate' → return this.repo.softDeleteByExternalId(externalId, this.provider);
    //   'noop'     → /* tombstone-preserving */ return null;
  }

  // ABSTRACT seams (NO body — a default body returning TCanonical is TS2322).
  // The projection-default subclass wires the default functions; a widened
  // subclass reimplements them (FK activation / typed-json narrow / null-coerce).
  protected abstract toCanonicalView(row: <Entity>IntegrationProjection): TCanonical;
  protected abstract buildWrite(record: TCanonical): <Entity>IntegrationWrite;
}
```
