# Junction pattern: association methods on canonical ports

## Goal

After this lands, generating `opportunity_contact.yaml` (or any junction) emits the four canonical fan-out methods on **both** parent services AND inside the junction service body — mirrored, paginated, composed `{ entity, link }` shape — without any hand-written association code. Consumer-visible change: `OpportunityService.attachContact(...)`, `OpportunityService.contactsList(...)`, `ContactService.addToOpportunity(...)`, `ContactService.opportunitiesList(...)` etc. all exist post-codegen. PR #363 (#61 PR A) snapshots will regenerate to contain both #358's relationship-composition methods and #60's junction fan-out on `contact.service.ts` — the implicit coexistence test.

## Out of scope

- **Junction controllers / DTOs / HTTP surface** — junctions remain non-addressable per #59 Q1.
- **`expose_on_parent: false` runtime semantics beyond skipping inject** — when set, the inject template emits nothing on that side; the junction service body is unaffected. No follow-up behavior (e.g. throwing in `attach`) is added.
- **Time-ordered cursor pagination** (`started_at desc`) — flagged as Open Q3 below; v1 paginates `<targets>List` by **right-entity id** matching #358's has_many cursor convention. No surrogate `id` exists on the junction table (composite PK), so junction rows themselves are not directly cursorable in v1.
- **Mutating fan-out for cross-domain pairings with layer-rule conflicts** — `opportunity_activity` is in scope (acceptance requires it), but if a future cross-domain pairing surfaces a layer-rule violation (e.g. write-from-a-passive-side rule), surface and stop; do not paper over.
- **Refactoring #358's entity-pipeline emission** — this leaf only injects; it does not modify `templates/entity/new/clean-lite-ps/service.ejs.t` or its clean-pipeline siblings.
- **Clean architecture pipeline** — `clean` variants are documented-broken per #61 §Discoveries (3 root causes, not introduced by this leaf). Smoke gate is **clean-lite-ps only**; `-clean` recipes are best-effort and not in `test-all`. Same "known gap" framing as #61.
- **Frontend / Electric-side composition** — backend service-method shape only.

## Architecture

### Three emission sites

```
opportunity_contact.yaml
  │
  ├─[hygen junction new]─► templates/junction/new/service.ejs.t
  │                          → OpportunityContactService.attach/detach/list/setPrimary
  │                          (composes { entity, link } via 2 single-table queries)
  │
  ├─[hygen junction new]─► templates/junction/new/_inject-parent-service.ejs.t  (left)
  │                          → injects into opportunity.service.ts (foreign file)
  │                          methods: attachContact, detachContact, contactsList, contactsSetPrimary
  │
  └─[hygen junction new]─► templates/junction/new/_inject-parent-service.ejs.t  (right)
                             → injects into contact.service.ts (foreign file)
                             methods: addToOpportunity, removeFromOpportunity,
                                      opportunitiesList, opportunitiesSetPrimary
```

### Cross-emission seam — first of its kind

**#60 is the first codegen-patterns emitter that writes onto a foreign entity's service file.** #358's entity-pipeline emission (#362, see `templates/entity/new/clean-lite-ps/service.ejs.t:100-148`) is **self-emission** — the entity's own YAML declares its own `relationships:` block and the entity's own service template writes the methods. Same for the legacy relationship pipeline at `templates/relationship/new/service.ejs.t:2` (`to: <%= outputPaths.service %>` — outputPaths point at the relationship's own service file, not a parent's). Neither establishes a cross-entity-write seam.

The mechanism #60 uses is **Hygen `inject: true`** templates — precedent in this repo: `templates/entity/new/backend/domain/_inject-index.ejs.t` (and 16 siblings). Each inject template specifies `to:` pointing at a foreign file, `inject: true`, an anchor (`before:` / `after:` / `append:`), and a `skip_if:` marker for idempotency.

Future emitters that need to cross-write (cross-entity policies, observability injections, etc.) follow the same seam. Document this in the spec's "Architecture" section header so it's discoverable.

### Method signatures (canonical, locked)

For pairing `between: [<L>, <R>]` with junction `<J>` (e.g. `<L>=opportunity`, `<R>=contact`, `<J>=opportunity_contact`):

**Left parent service** (`<L>Service`, e.g. `OpportunityService`):

```typescript
async attach<RPascal>(
  <l>Id: string, <r>Id: string,
  link?: Partial<JunctionLinkInput>,
): Promise<<JPascal>>;
async detach<RPascal>(<l>Id: string, <r>Id: string): Promise<void>;
async <rPlural>List(
  <l>Id: string, opts?: { cursor?: string; limit?: number },
): Promise<Array<{ entity: <RPascal>; link: <JPascal> }>>;
async <rPlural>SetPrimary(<l>Id: string, <r>Id: string): Promise<void>;
```

**Right parent service** (`<R>Service`, e.g. `ContactService`) — same shape, swapped, with verb flips for natural reading:

```typescript
async addTo<LPascal>(<r>Id: string, <l>Id: string, link?: Partial<JunctionLinkInput>): Promise<<JPascal>>;
async removeFrom<LPascal>(<r>Id: string, <l>Id: string): Promise<void>;
async <lPlural>List(<r>Id: string, opts?: { cursor?: string; limit?: number }): Promise<Array<{ entity: <LPascal>; link: <JPascal> }>>;
async <lPlural>SetPrimary(<r>Id: string, <l>Id: string): Promise<void>;
```

All eight methods on parent services delegate to the **junction service** (4 methods, body-emitted in `service.ejs.t`):

```typescript
async attach(<l>Id: string, <r>Id: string, link?: <JPascal>LinkInput): Promise<<JPascal>>;
async detach(<l>Id: string, <r>Id: string): Promise<void>;
async listAssoc(
  side: 'left' | 'right', anchorId: string,
  opts?: { cursor?: string; limit?: number },
): Promise<Array<{ entity: <LPascal> | <RPascal>; link: <JPascal> }>>;
async setPrimary(<l>Id: string, <r>Id: string): Promise<void>;
```

**Implementation revision — `list` → `listAssoc`.** `BaseService` already
declares `list(options?: unknown): Promise<TEntity[]>` with an incompatible
return shape (`TEntity[]` vs `Array<{ entity, link }>`). TypeScript will
reject a same-name override. Renaming the junction service helper to
`listAssoc` keeps the canonical parent-side names (`<rPlural>List(...)`)
untouched while removing the override collision. Spec-time name was `list`;
implementation discovered the conflict.

`list` implementation — two single-table queries, no Drizzle `with:`:

```
const links = await this.repository.findBy<Anchor>Id(anchorId, opts);   // existing finder from #359
const targetIds = links.map(l => l[<oppositeCol>]);
const targets = await this.<opposite>Repo.findByIds(targetIds);           // BaseRepository.findByIds
const byId = new Map(targets.map(t => [t.id, t]));
return links.map(link => ({ entity: byId.get(link[<oppositeCol>])!, link }));
```

Cursor column for v1: **right-entity `id`** on the right-FK column (matches #358 has_many shape). Composite-PK junction rows have no surrogate id; ordering by `started_at desc` is Open Q3.

`JunctionLinkInput` = `Pick<<JPascal>, 'is_primary' | 'started_at' | 'ended_at' | 'sourced_from' | 'confidence' | 'matched_at' | 'role'>` minus `role` when `hasRole === false`.

### Opt-out heuristic

Junction YAML accepts a new optional key:

```yaml
expose_on_parent:
  left: false   # suppress left parent service injection (and module wiring)
  right: false  # suppress right parent service injection
```

Defaults: both `true`. Junction service body is **always** emitted regardless. **Bundle the #58 schema patch into this PR** (~6 LOC addition to `src/schema/junction-definition.schema.ts` after the existing `.strict()` block — same pattern as `temporal`/`sourced`). If the patch exceeds ~10 LOC, halt and reconsider.

### Ordering contract

Entity pipeline emits first (writes the parent service file with `force: true`), junction pipeline injects second. The smoke harness already invokes in this order (`codegen entity new --all` then `codegen junction new --all`). Document this as a contract:

- In a new top-comment in `test/smoke/run-smoke-junction.ts` after the existing arg-parsing block.
- In `src/cli/commands/junction.ts` `JunctionNewCommand` execute path: if the inject template's anchor (`// Inherited from BaseService:`) is absent in the target file, Hygen will produce a clear error — let it propagate, do not swallow. Acceptable failure mode for users who run junction codegen before entity codegen.

## Files

### Create

| File | Purpose |
|---|---|
| `templates/junction/new/_inject-parent-service-clp-{left,right}.ejs.t` | Hygen inject template pair for clean-lite-ps parent services (one per side; Hygen requires distinct static `to:` per file). `to:` resolves to `<srcRoot>/modules/<parentPlural>/<parent>.service.ts`. `inject: true`, `before: "// Inherited from"` (substring anchor — the rendered marker is `// Inherited from <ServiceBaseClass>:` and `<ServiceBaseClass>` is family-aware: `BaseService`, `SyncedEntityService`, `ActivityEntityService`, etc. The substring `// Inherited from` is stable across all families). `skip_if: "<%= injectionMarkerLeft \| injectionMarkerRight %>"` where the marker is `// junction:<name>:left-fan-out` / `// junction:<name>:right-fan-out`. Emits property-injection (`@Inject(forwardRef(() => <JService>)) private readonly <jCamel>Service!: <JService>`) PLUS the four methods + the per-junction marker comment. **Property injection chosen over constructor injection** — Hygen anchoring on a constructor parameter list is brittle (positional commas); property injection only requires the stable `// Inherited from` anchor. `forwardRef` resolves the parent↔junction module cycle (parent imports junction module to wire the service; junction module imports parent modules for repo DI). |
| `templates/junction/new/_inject-parent-service-import-clp-{left,right}.ejs.t` | Sibling inject pair: adds `import { forwardRef } from '@nestjs/common'; import { <JService>, <JPascal>LinkInput } from '...'; import type { <JPascal>, <OppositeEntity> } from '...';` after the line containing `from '@nestjs/common';` in the parent service file. `skip_if: "<%= classNames.service %> }"` (the `}` disambiguates from the property name). |
| `templates/junction/new/_inject-parent-module-clp-{left,right}.ejs.t` | Sibling inject pair: adds `forwardRef(() => <JPascal>Module)` to the parent module's `imports:` array. Anchor: `after: "    DatabaseModule,"` (stable indented line in clean-lite-ps parent modules). ## Open questions (resolved at spec-approval time)

- **Q1 — Naming for `setPrimary` and `list` on the right side.** Spec ships flat-noun (`contactsSetPrimary`, `opportunitiesList`, `lPlural`/`rPlural` style). Verb-flip only applied where the plan example explicitly does (`attach`/`add`, `detach`/`remove`). **Resolved: approved by team-lead.**
- **Q2 — `link.role` required when junction declares non-nullable role enum.** TS-level required via conditional `Pick`. If the implementer finds the conditional type painful, halt and surface. **Resolved: required by default; halt-and-surface escape valve.**
- **Q3 — Time-ordered cursor pagination on `<targets>List`.** Deferred to v2. v1 paginates by right-entity `id` only. **Resolved: deferred.**
- **Q4 — Inject-anchor stability assertion in smoke.** Added to the smoke harness: assert the literal `// Inherited from BaseService:` is present in every emitted parent service file. Cheap anti-regression. **Resolved: include in this PR (folded into the "Modify `run-smoke-junction.ts`" line in Files §).**
- **Q5 — Junction service body emission under `expose_on_parent: { left: false, right: false }`.** Yes — junction service body always emits regardless. Consumers can call the junction service directly even with both sides opted out. **Resolved: junction service always emits the 4 methods.**
- **Bundling — `expose_on_parent` schema patch on `src/schema/junction-definition.schema.ts`.** Bundled into this PR. Escape valve: if patch >10 LOC, halt and reconsider deferring. **Resolved: bundled.**

## Risks

| Risk | Mitigation |
|---|---|
| **(a) #358 emitter and #60 inject both write to same parent service file → ordering / merge.** | Entity pipeline runs first (`codegen entity new --all` then `codegen junction new --all` in smoke); `force: true` on entity, `inject: true` on junction. Inject anchor (`// Inherited from BaseService:`) is stable across #362. `skip_if` is **per-junction unique** (`// junction:<%= name %>:<%= side %>-fan-out`) so multiple junctions on one parent service do NOT silently skip after the first. Generic marker (`// junction-fan-out`) would be a bug — explicitly avoided. |
| **(b) Cross-entity-write seam is novel.** | Documented as the architectural precedent in this spec. Inject templates use vanilla Hygen — no new infrastructure. Precedent files (`templates/entity/new/backend/**/_inject-*.ejs.t`) prove the seam works in this codebase. |
| **(c) User runs `codegen junction new` before `codegen entity new`.** | Hygen surfaces a missing-anchor error. Acceptable failure mode. Document in `run-smoke-junction.ts` top comment and in the `JunctionNewCommand` execute body's error handling — propagate the Hygen error verbatim, do not catch. |
| **(d) Schema patch widens #58's contract mid-flight.** | Patch is ~6 LOC. Test added to `src/__tests__/schema/junction-definition.test.ts` for the new field. If implementer finds the patch >10 LOC, halt and reconsider deferring to a #58-style follow-up. |
| **(e) Cross-domain pairing `opportunity_activity` may have layer-rule conflicts.** | Smoke covers it; if a layer violation surfaces, surface and stop (do not paper over). |
| **(f) #363 (PR A) snapshots regenerate with churn.** | Expected — that's the implicit coexistence test. #363's `bun test --update-snapshots` flow handles it; document in this PR's description that #363 will need a snapshot regen commit after #60 merges. |
