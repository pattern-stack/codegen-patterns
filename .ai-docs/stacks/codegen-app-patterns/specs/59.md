# Junction pattern: Hygen templates (table + repo + service shell)

## Goal

Land the **structural** Hygen templates for the Junction pattern in `pattern-stack/codegen-patterns` as a single PR. After this issue merges, `codegen junction new --yaml junctions/opportunity_contact.yaml` produces a compiling Drizzle table + typed CRUD repo + service shell for any junction pairing — intra-domain OR cross-domain — and the generated files merge cleanly into the existing barrel graph at the correct import depth.

What this leaf ships:

1. A **table template** emitting the Drizzle schema for the junction pairing: composite primary key on the two FK columns plus the role column (typed from the per-pairing enum), the six `BaseJunctionFields` columns from #58, FK `references()` with `onDelete` per pairing, and a `<plural>Relations = relations(...)` const for the extension path (parity with how Relationship and entity templates emit relations metadata).
2. A **repo template** emitting a NestJS `@Injectable()` class that extends `BaseRepository<JunctionEntity>` — inheriting `findById / findByIds / list / count / exists / create / update / delete / upsertMany` — and adding two **pairing-aware finders** (`findByLeftId` / `findByRightId`, named per the canonical endpoint columns). The repo is single-table CRUD only; no cross-entity composition lives here.
3. A **service shell** that extends `WithAnalytics(BaseService<JunctionRepository, JunctionEntity>)` with the canonical constructor signature, EVENT_BUS @Optional() injection, `entityName` override, and pass-through `findByLeftId` / `findByRightId` query methods. **No fan-out attach/detach/list/setPrimary methods.** Those land in #60 once codegen-patterns#358 establishes the service-method emission machinery — see [Scope carve-out](#scope-carve-out).
4. A **`codegen junction`** CLI noun mirroring `codegen relationship` — `junction new <file>`, `junction new --all`, `junction list`, all driven by `detectYamlType() === 'junction'` (introduced by #58).
5. **Barrel integration** so junction modules merge into `GENERATED_MODULES` and `<generated>/schema.ts` at the right depth on BOTH architecture pipelines (`clean` and `clean-lite-ps`). The `01bb917` import-depth regression is pinned by a dedicated smoke assertion.
6. **Hygen-driven smoke fixtures** exercising both pipelines × both pairing topologies (intra-domain + cross-domain) — four code paths total — through `bunx tsc --noEmit` plus targeted greps that the canonical class shape is present.

## Scope carve-out

This leaf was substantially descope'd by a recent design discussion (encoded in the plan's `architectural_notes.cross_entity_access` block and the conductor's prompt). Read carefully:

- **EMIT** the three structural artifacts: junction table, junction repo, junction service **shell**.
- **DO NOT EMIT** the fan-out association methods:
  - `OpportunityService.attachContact / detachContact / contacts.list / contacts.setPrimary`
  - `ContactService.addToOpportunity / removeFromOpportunity / opportunities.list / opportunities.setPrimary`
  - `OpportunityContactService.attach / detach / list / setPrimary`
- The service shell **must carry the canonical class shape #60 will later extend** — same constructor signature, same DI tokens, same export shape, same `entityName` override, same `WithAnalytics(BaseService<...>)` mixin. The body of fan-out methods does not exist yet in this leaf.

If during implementation you find yourself adding `attach()` / `detach()` / `list()` / `setPrimary()` method bodies to the service shell, **stop and surface**. That work is #60 and is blocked on codegen-patterns#358. The plan's `external_dependencies` block lists the rationale.

The **interim workaround** for wave-1 consumers (per plan): hand-write the fan-out methods against the `architectural_notes.cross_entity_access.canonical_shape` contract. The hand-written code is the executable spec for codegen-patterns#358; when #358 merges, #60 generates the same shape mechanically and replaces the hand-written methods.

## Architecture

### Pipeline shape — mirror Relationship, not Entity

Junction is structurally a Relationship variant (two endpoints, optional role enum, temporal + sourcing metadata, single junction table). The Relationship pipeline at `templates/relationship/new/` is the obvious precedent: independent generator with its own `prompt.js`, its own table/repo/service/module/index templates, invoked via `codegen relationship new <yaml>`.

Junction mirrors this structurally — a NEW generator at `templates/junction/new/` with its own `prompt.js`, NOT a contribution into the entity pipeline. Rationale:

1. **Disjoint YAML discriminator.** Junction files are top-level `pattern: Junction` (per #58 Q1 resolution, ship the literal). Entity files are top-level `entity:`. Relationship files are top-level `relationship:`. Each gets its own pipeline.
2. **The CLI verb is already orthogonal.** `codegen entity` operates on `entities/*.yaml`; `codegen relationship` operates on `relationships/*.yaml`. A new `codegen junction` noun operating on `junctions/*.yaml` is consistent.
3. **No contribution-into-entity coupling.** Junction does NOT add columns to either of the two parent entities (no FK columns on `accounts.contacts_id` — the FKs live on the junction table). Entity templates have no business knowing about junction pairings until #60 emits cross-entity association methods, at which point #60 is responsible for the cross-entity emission, not this leaf.

The new generator directory layout mirrors `templates/relationship/new/`:

```
templates/junction/new/
  prompt.js                  # loads junction YAML via loadJunctionFromYaml (#58), derives locals
  entity.ejs.t               # Drizzle table + enums + relations() const
  repository.ejs.t           # BaseRepository<JunctionEntity> + pairing-aware finders
  service.ejs.t              # WithAnalytics(BaseService<...>) shell — NO fan-out methods
  module.ejs.t               # NestJS @Module wiring the three above
  index.ejs.t                # barrel re-exports
```

**Deliberately omitted from this leaf's generator output** (vs Relationship's):

- `controller.ejs.t`, `dto/`, `use-cases/` — junctions are not directly addressable HTTP resources. They are accessed via the canonical port of one of the two parent entities (per `canonical_shape.junction_association_placement`). Emitting a junction controller would be premature surface area; if it turns out to be needed, it lands as a follow-up. Surface this as Open Question Q1.
- Declarative `queries:` block emission with arbitrary user fields — **deferred to follow-up**. v1 hardcodes the two pairing-aware finders (`findBy<Left>Id`, `findBy<Right>Id`) because every junction needs them. User-declared `queries:` (from the YAML's `queries:` block) can land in a v2 if consumers actually use them; the Relationship template already shows how (`prompt.js` `processQueries`). Surface as Open Question Q2.

### On-disk output paths — match the existing layout per pipeline

Output paths follow whatever `entityFilePaths()` in `src/cli/shared/barrel-generator.ts:189-225` expects, since that function computes the import paths the barrel uses. The junction module + entity files must land where the barrel will look for them; otherwise the `01bb917` import-depth bug regresses.

**clean-lite-ps:**

```
<srcRoot>/modules/<plural>/<name>.entity.ts
<srcRoot>/modules/<plural>/<name>.repository.ts
<srcRoot>/modules/<plural>/<name>.service.ts
<srcRoot>/modules/<plural>/<plural>.module.ts
<srcRoot>/modules/<plural>/index.ts
```

`<srcRoot>` defaults to `src` (matches `clean-lite-ps/prompt-extension.js`), overridable via `paths.backend_src` in `codegen.config.yaml`. `<name>` is the snake_case junction name (e.g. `opportunity_contact`); `<plural>` is `pluralize(name)` unless `table:` is declared on the junction YAML.

**clean (full Clean Architecture):**

```
<backendSrc>/domain/<plural>/<name>.entity.ts
<backendSrc>/domain/<plural>/<name>.repository-interface.ts
<backendSrc>/infrastructure/persistence/drizzle/<pluralKebab>.schema.ts
<backendSrc>/infrastructure/persistence/drizzle/<pluralKebab>.repository.ts
<backendSrc>/application/<plural>/<name>.service.ts
<backendSrc>/infrastructure/modules/<pluralKebab>.module.ts
```

`<backendSrc>` defaults to `app/backend/src` per `resolveBackendSrc()`. Junction's clean-pipeline layout mirrors the entity template's layout (`templates/entity/new/backend/{domain,application,database,modules}/...`) so that `entityFilePaths()` returns paths that exist on disk.

**Critical:** confirm against `barrel-generator.ts:189-225` that the chosen file layout matches what `entityFilePaths()` returns. The cleanest implementation reuses `entityFilePaths()` directly (junctions are treated as peer modules per `collectRelationships()` at `barrel-generator.ts:156-169` — which already calls `entityFilePaths()`). If the layout diverges, EITHER fix the templates to match `entityFilePaths()` OR extend `entityFilePaths()` to accept a discriminator and emit junction-specific paths — the FIRST is simpler unless `01bb917`'s logic depends on junction-specific structure.

### Drizzle table emission — what changes vs Relationship

Read `templates/relationship/new/entity.ejs.t` as the structural baseline. The junction template diverges in five places:

1. **No type enum.** Relationships have `types: [employed_by, mentors, ...]`; junctions have a `role` enum declared in the junction YAML's `fields:` block (per #58 Q2 resolution: ship single `role` field). The role enum becomes a `pgEnum` whose values come from the YAML's `fields.role.choices`, not from a top-level `types:` config block.
2. **`BaseJunctionFields` columns** — six fixed columns emitted unconditionally: `is_primary boolean`, `started_at timestamp`, `ended_at timestamp`, `sourced_from text`, `confidence numeric(5,4)`, `matched_at timestamp`. Imported from `@pattern-stack/codegen/runtime/junction/base-junction-fields` (TODO confirm runtime export location — see Open Question Q3) so the schema file matches the registry-declared shape from #58. **No `valid_from / valid_to / is_current` temporal columns** — those are Relationship-pattern shape; #58's `BaseJunctionFields` uses `started_at / ended_at` instead. **No `source pgEnum`** — junctions track provenance via `sourced_from text` (free-form) and `confidence numeric(5,4)`, not via an enum.
3. **Composite primary key on `(left_fk, right_fk)`** — replacing the Relationship template's `id: uuid('id').primaryKey().defaultRandom()`. The junction's identity IS the pairing. Worked example: `primaryKey({ columns: [table.opportunityId, table.contactId] })` emitted in the third positional arg of `pgTable`. This is the load-bearing topology difference — junctions don't have a surrogate id (per ADR-031 and #58's BaseJunctionFields shape: there is no `id` column listed). Surface as Open Question Q4 — pending #58 confirming whether a surrogate `id` is emitted.
4. **`temporal` / `sourced` toggles act as suppressors, not adders** (per #58 Q4 resolution if kept as opt-outs). Default `temporal: true` keeps `started_at + ended_at`; `temporal: false` suppresses both. Default `sourced: true` keeps `sourced_from + confidence + matched_at`; `sourced: false` suppresses those three. If #58 lands with `temporal/sourced` dropped from the schema, this leaf drops the conditional branches and emits unconditionally.
5. **`<plural>Relations = relations(...)` extension-path metadata.** Same emission as Relationship's `entity.ejs.t` and entity's `backend/database/schema.ejs.t:224-244`. The junction is on one side of two `one(...)` relations pointing at the two parent tables. The relations const is extension-path metadata (`relations()` enables hand-written `db.query.X.findMany({ with: ... })` for backend admin queries) — generated code does NOT use it. Per `architectural_notes.cross_entity_access` and cgp-62 r4: keep emitting. Diagram:

```mermaid
flowchart LR
  Y[opportunity_contact.yaml<br/>pattern: Junction<br/>between: [opportunity, contact]] -->|loadJunctionFromYaml #58| P[prompt.js<br/>derive locals]
  P -->|entity.ejs.t| T[opportunity_contacts table<br/>FK×2 + role + BaseJunctionFields<br/>composite PK]
  P -->|entity.ejs.t| R[opportunityContactsRelations<br/>one(opportunities), one(contacts)<br/>extension metadata]
  P -->|repository.ejs.t| RP[OpportunityContactRepository<br/>extends BaseRepository<br/>+ findByOpportunityId, findByContactId]
  P -->|service.ejs.t| S[OpportunityContactService<br/>extends WithAnalytics&#40;BaseService&#41;<br/>SHELL — no fan-out]
  P -->|module.ejs.t| M[OpportunityContactsModule]
  T & R & RP & S & M -->|barrel-generator| B[GENERATED_MODULES<br/>schema.ts re-exports]
  P -.fan-out methods.-> X[#60 + codegen-patterns#358<br/>NOT in this leaf]
```

### Repo template — what's emitted, what isn't

The repo extends `BaseRepository<JunctionEntity>` (re-using the runtime base class at `runtime/base-classes/base-repository.ts:45`), inheriting `findById / findByIds / list / count / exists / create / update / delete / upsertMany`. The repo template emits:

- The `@Injectable()` decorator and class declaration.
- The `readonly table = <plural>;` binding.
- A `behaviors: BehaviorConfig` override with `{ timestamps: true, softDelete: false, userTracking: false }` — junctions track temporal validity via `started_at / ended_at`, NOT via `deleted_at`; `is_primary` flips replace soft-delete semantics. Surface as Open Question Q5.
- The standard `constructor(@Inject(DRIZZLE) db: DrizzleClient) { super(db); }`.
- **Two pairing-aware finders**, hardcoded into the template (not derived from the YAML's `queries:` block):
  - `findBy<Left>Id(<leftCamel>Id: string, opts?: { cursor?: string; limit?: number }): Promise<JunctionEntity[]>` — paginated per `architectural_notes.cross_entity_access.canonical_shape.pagination_default`.
  - `findBy<Right>Id(<rightCamel>Id: string, opts?: { cursor?: string; limit?: number }): Promise<JunctionEntity[]>` — same shape, opposite FK.

Method names derive from the endpoint column names (e.g. `between: [opportunity, contact]` → `findByOpportunityId / findByContactId`). The pagination opts shape **must match the convention adopted by codegen-patterns#358** — if #358 ships pagination via `{ cursor, limit }`, this leaf does the same; if #358 picks something different, this leaf adopts whatever #358 picks. Surface as Open Question Q6: if #358 has not yet picked a pagination shape by the time #59 implements, this leaf ships `{ cursor?, limit? }` as the canonical shape per cgp-62 r4 and tags the method bodies with a `// FIXME: align with codegen-patterns#358 pagination shape if it differs` comment.

The pairing-aware finders **do not** compose entities from the other side of the pairing. They return raw junction rows. Composition (the `{ entity, link }` join-shape DTO from `canonical_shape.list_association_return_shape`) lives in the fan-out service methods #60 emits. The repo stays single-table.

The TODO `// TODO: Add relationship-specific query methods here.` block from Relationship's `repository.ejs.t:50` is dropped — junctions don't need an open-ended TODO; the canonical method set is fully determined by the pairing.

### Service shell — canonical class shape #60 will extend

The service shell mirrors `templates/relationship/new/service.ejs.t` exactly, with the body trimmed of any relationship-specific declarative-query pass-throughs. It contains:

```typescript
@Injectable()
export class <Pascal>Service extends WithAnalytics(
  BaseService<<Pascal>Repository, <Pascal>>,
) {
  protected override readonly entityName = '<snake>';

  /** Injected by NestJS when EventsModule is registered. */
  @Optional() @Inject(EVENT_BUS)
  protected override eventBus: any = undefined;

  constructor(protected override readonly repository: <Pascal>Repository) {
    super(repository);
  }

  // Pairing-aware pass-throughs (mirror the repo's two finders so use-cases
  // and #60's fan-out methods both delegate through the service layer per
  // the analytics/events uniformity rule from relationship's service.ejs.t).
  async findBy<Left>Id(
    <leftCamel>Id: string,
    opts?: { cursor?: string; limit?: number },
  ): Promise<<Pascal>[]> {
    return this.repository.findBy<Left>Id(<leftCamel>Id, opts);
  }

  async findBy<Right>Id(
    <rightCamel>Id: string,
    opts?: { cursor?: string; limit?: number },
  ): Promise<<Pascal>[]> {
    return this.repository.findBy<Right>Id(<rightCamel>Id, opts);
  }

  // ─────────────────────────────────────────────────────────────────────────
  // Fan-out association methods (attach/detach/list/setPrimary) are NOT
  // emitted here. They land via junction-association-codegen
  // (pattern-stack/dealbrain-integrations#60) once
  // pattern-stack/codegen-patterns#358 establishes the service-method
  // emission machinery.
  //
  // Until #60 lands, consumers hand-write fan-out methods in their own
  // service files against the canonical shape documented in
  // .ai-docs/stacks/codegen-app-patterns/specs/cgp-62.md (see
  // architectural_notes.cross_entity_access.canonical_shape in the stack
  // plan). The hand-written methods are the executable spec for #60.
  // ─────────────────────────────────────────────────────────────────────────

  // Inherited from BaseService:
  //   findById, findByIds, list, count, exists, create, update, delete
}
```

The big comment block is **load-bearing**. It's the marker future implementers (human or #60's agent) read to know where to wire fan-out methods, and it's the marker code reviewers use when they ask "where's the attach method?". Do not omit the comment block, even though it's prose-only.

The `<Left>` / `<Right>` placeholders derive from the YAML's `between: [a, b]` endpoint names. For `between: [opportunity, contact]`, `<Left>` = `Opportunity`, `<Right>` = `Contact`.

### CLI noun — `codegen junction`

Mirror `src/cli/commands/relationship.ts` (394 lines). Three commands, default export as a `NounModule`:

- `JunctionNewCommand` — `codegen junction new <yaml>` or `--all`, with `--dry-run`, `--force`, `--json`, `--cwd`, `--config` options. ### Barrel integration — extending `regenerateBarrels()`

`src/cli/shared/barrel-generator.ts` already handles junction-from-relationship modules via `collectRelationships()` at L156-169 — it discovers `relationships/*.yaml`, treats each as a peer module of regular entities, and merges them into `GENERATED_MODULES` + `<generated>/schema.ts`. The 01bb917 commit fixed an import-depth bug at this exact seam.

This leaf needs a parallel **`collectJunctions()`** function:

```typescript
function collectJunctions(junctionsDir: string): EntityInfo[] {
  const files = listJunctionYamls(junctionsDir);   // filter by detectYamlType()==='junction'
  const junctions: EntityInfo[] = [];
  for (const file of files) {
    const result = loadJunctionFromYaml(file);
    if (!result.success) continue;
    const def = result.definition;
    const name = deriveJunctionName(def);          // see Open Question Q8: derive from between[] or from explicit `name:`
    const plural = def.table ?? pluralize(name);
    junctions.push({ name, plural });
  }
  junctions.sort((a, b) => a.name.localeCompare(b.name));
  return junctions;
}
```

`regenerateBarrels()` at L294-350 is extended to merge in `collectJunctions(<cwd>/junctions)` alongside `collectEntities()` and `collectRelationships()`. The merged list goes through the same `entityFilePaths()` → `buildModulesBarrel()` → `buildSchemaBarrel()` pipeline. Junction modules thus appear in `GENERATED_MODULES` and `<generated>/schema.ts` exactly the same way regular and relationship modules do; no new code path, no special-casing in the barrel content.

**01bb917 regression prevention:** the smoke test asserts `bunx tsc --noEmit` succeeds on the generated tmp project after junction codegen runs. A failed import resolution at the wrong depth surfaces as a TS2307 (`Cannot find module './<wrong-path>'`). The smoke harness already catches this for the entity + relationship pipelines; extending it to cover junctions closes the loop. See [Tests](#tests).

### Pipeline coverage — both `clean` and `clean-lite-ps` must work

The plan explicitly requires both pipelines. Two ways this could go wrong:

1. **Path divergence.** The repository.ejs.t / entity.ejs.t output paths differ per architecture. `templates/relationship/new/` currently hardcodes the clean-lite-ps layout (`src/modules/<plural>/<name>.entity.ts`) in its `prompt.js:534-544` `outputPaths`. This is a known limitation of the Relationship pipeline that the implementer of THIS leaf must NOT inherit — junction templates must derive output paths from `resolveArchitecture(ctx)` the same way `entityFilePaths()` does. Concretely: junction's `prompt.js` reads `paths.backend_src` and `generate.architecture` from `codegen.config.yaml` and computes paths conditionally. Implementer reads `templates/entity/new/prompt.js` + `clean-lite-ps/prompt-extension.js` for the precedent (entity templates already do this branching; junction copies the pattern).

2. **Base-class import paths differ.** Clean-lite-ps imports `BaseRepository` from `@shared/base-classes/base-repository` (alias resolved via `tsconfig.json`'s `paths`); clean uses the same alias. The junction template uses the same alias unconditionally. If the alias is missing in a consumer project, `codegen project init` is responsible for adding it (already done — verified by the `relationship/new/repository.ejs.t:11` import working in the smoke).

Both pipelines run through the same smoke harness, so divergence surfaces as a tsc error on one architecture but not the other. See [Tests](#tests) for the matrix.

### Hygen invocation — extend `invokeHygen` helpers

`src/cli/shared/hygen.ts` is currently 100 lines and exports two convenience wrappers: `invokeEntityNew` and `invokeRelationshipNew`. This leaf adds a third:

```typescript
export function invokeJunctionNew(absoluteYamlPath: string, cwd?: string): HygenResult {
  return invokeHygen({
    generator: 'junction',
    action: 'new',
    args: ['--yaml', absoluteYamlPath],
    cwd,
  });
}
```

The `--bun` flag in the shared `invokeHygen` body is preserved verbatim — junction's `prompt.js` does `await import('@pattern-stack/codegen/utils/yaml-loader.js')` (or the local equivalent), which requires Bun's `.js`-to-`.ts` resolver per the comment block at `hygen.ts:6-19`. If junction's prompt.js does not need to import any TypeScript modules (it might if it reuses `BASE_JUNCTION_FIELD_NAMES` from `src/patterns/library/base-junction-fields.ts`), `--bun` is still needed for consistency and to keep the regression test at `src/__tests__/cli/hygen.test.ts` green.

### Coexistence — junction + entity `relationships:` block in same fixture

The plan acceptance criteria for the SIBLING leaf (`junction-test-fixtures`, #61) include "same entity YAML carries both `pattern: Junction` and a `relationships:` block." For THIS leaf, the more limited requirement is: codegen against a junction YAML referencing entities that ALSO have a `relationships:` block does not collide on names, on import paths, or on barrel entries. The smoke fixture set covers this with one fixture pairing (`opportunity_contact`) where the parent entities carry `relationships:` blocks (mirroring the cgp-62 fixture set). See [Tests](#tests).

### Assertion helpers

```typescript
// test/smoke/run-smoke.ts — additions near runSmoke().

type Scenario = 'default' | 'relationship' | 'junction' | 'junction-cross-domain';

function assertJunctionEmission(
  generatedSrc: string,
  scenario: 'junction' | 'junction-cross-domain',
  architecture: 'clean' | 'clean-lite-ps',
): void {
  const reads = (p: string): string => fs.readFileSync(path.join(generatedSrc, p), 'utf8');

  const { junctionName, pluralName, leftEnt, rightEnt } = scenario === 'junction'
    ? { junctionName: 'opportunity_contact', pluralName: 'opportunity_contacts', leftEnt: 'opportunity', rightEnt: 'contact' }
    : { junctionName: 'opportunity_activity', pluralName: 'opportunity_activities', leftEnt: 'opportunity', rightEnt: 'activity' };

  const junctionDir = architecture === 'clean-lite-ps'
    ? `src/modules/${pluralName}`
    : `app/backend/src/domain/${pluralName}`;       // (clean) — implementer confirms against entityFilePaths()

  const entityFile = reads(`${junctionDir}/${junctionName}.entity.ts`);
  assertContains(entityFile, /primaryKey\(\{\s*columns:\s*\[table\.\w+Id,\s*table\.\w+Id\]/);
  assertContains(entityFile, /is_primary|isPrimary/);
  assertContains(entityFile, /started_at|startedAt/);
  assertContains(entityFile, /ended_at|endedAt/);
  assertContains(entityFile, /sourced_from|sourcedFrom/);
  assertContains(entityFile, /confidence:\s*numeric/);
  assertContains(entityFile, /matched_at|matchedAt/);
  assertContains(entityFile, new RegExp(`${pluralName}Relations\\s*=\\s*relations\\(${pluralName}`));

  const repoFile = reads(`${junctionDir}/${junctionName}.repository.ts`);
  assertContains(repoFile, /extends BaseRepository<\w+>/);
  assertContains(repoFile, new RegExp(`findBy${pascal(leftEnt)}Id\\s*\\(`));
  assertContains(repoFile, new RegExp(`findBy${pascal(rightEnt)}Id\\s*\\(`));
  assertContains(repoFile, /cursor\?:\s*string;\s*limit\?:\s*number/);

  const svcFile = reads(`${junctionDir}/${junctionName}.service.ts`);
  assertContains(svcFile, /extends WithAnalytics\(\s*BaseService</);
  assertContains(svcFile, /protected override readonly entityName/);
  assertContains(svcFile, /@Optional\(\)\s*@Inject\(EVENT_BUS\)/);
  assertContains(svcFile, /Fan-out association methods.*are NOT\s+\/\/ emitted here/s);
  // Anti-regression: fan-out methods MUST NOT appear in this leaf's output.
  assertAbsent(svcFile, /async\s+attach\s*\(/);
  assertAbsent(svcFile, /async\s+detach\s*\(/);
  assertAbsent(svcFile, /async\s+setPrimary\s*\(/);
}

function assertAbsent(haystack: string, needle: RegExp): void {
  if (needle.test(haystack)) {
    throw new Error(
      `Smoke assertion failed: expected NOT to find ${needle} in generated output. ` +
      `This leaf must not emit fan-out association methods — they land via #60.`,
    );
  }
}
```

## Open questions

These should be resolved during Gate-1 review (or punted to implementer with an explicit default).

- **Q1 — Junction controller emission.** Should the junction generator emit a `controller.ejs.t` (mirroring `relationship/new/controller.ejs.t`)? Junctions are typically reached via the parent entity's port, but admin tools / debugging surface might want direct CRUD access to junction rows. Default: NO controller in v1 (junctions are not addressable HTTP resources); add via follow-up if a consumer surfaces a need. Flag for reviewer.

- **Q2 — Declarative `queries:` block on junction YAML.** The Relationship template processes a `queries:` block at `prompt.js:254-317` and emits methods. Should junction's v1 do the same, or hardcode only the two pairing-aware finders? Default: hardcode only the two finders (every junction needs them, almost no junctions need anything more in early use). If a junction YAML carries a `queries:` block, v1 ignores it with a warning. Flag for reviewer.

- **Q3 — Runtime import path for `BaseJunctionFields`.** #58 places `BaseJunctionFields` in `src/patterns/library/base-junction-fields.ts`. Generated code (consumer-side) needs to import the constant — what's the resolvable import path? Options: (a) re-export from a `@pattern-stack/codegen/runtime/junction/base-junction-fields` barrel and the consumer's `tsconfig.json` `paths` aliases resolve it; (b) inline the six column emissions in the template itself (no runtime import). Default: ship (b) — the template renders the columns directly using the field names from `BASE_JUNCTION_FIELD_NAMES`. Inlining is fragile only if the field set drifts, and #58 has unit tests pinning the field set. Flag for reviewer.

- **Q4 — Composite PK vs surrogate `id`.** #58's `BaseJunctionFields` shape does NOT include `id`. Confirm with #58 reviewer (Q3 there): does the table have a surrogate `id` UUID column, or is the PK purely `(left_fk, right_fk)`? Default: composite PK on `(left_fk, right_fk)`, no surrogate id. If a surrogate id is needed (e.g. for foreign keys FROM other tables pointing into the junction), add it; but that's a YAGNI case for v1. Flag for reviewer.

- **Q5 — `softDelete` behavior on junction rows.** Junctions track temporal validity via `started_at / ended_at`. Removing a row could mean: (a) `delete()` (hard delete); (b) update `ended_at = now()` (lifecycle close). The base `BaseRepository.delete()` does whichever `behaviors.softDelete` says. This leaf defaults `softDelete: false`. Surface for reviewer — if temporal closure is the canonical semantic, the service shell could expose `closeAt(id, ts)` as a sibling of inherited `delete()`. Default: ship `softDelete: false` and let #60 add `endLink(...)` / similar if needed. Flag.

- **Q6 — Pagination shape consistency with codegen-patterns#358.** Per cgp-62 r4 the canonical shape is `{ cursor?, limit? }`. #358 has not yet implemented its pagination convention; if it ships with a different shape (e.g. `{ offset, limit }`), the two pairing-aware finders here must align. Default: ship `{ cursor?, limit? }` now and tag the finder bodies with a `// FIXME: align with codegen-patterns#358 if pagination shape diverges` comment. Flag.

- **Q7 — Junctions directory naming.** #58's fixtures live under `test/fixtures/junctions/` — confirm the runtime directory in consumer projects is also `junctions/` (matching `relationships/` for the relationship pipeline). Default: `junctions/` (consistent with fixture location and Relationship precedent). Flag.

- **Q8 — Junction `name` derivation.** When a junction YAML carries no explicit `name:`, derive from `between: [a, b]` as `<a>_<b>` (sorted? insertion order?). Default: insertion order, matching the `between` array order — `between: [opportunity, contact]` → `name: opportunity_contact`. If sorted alphabetically, the human-readable name doesn't always match consumer intuition. Flag.

- **Q9 — Existing dirty working tree in worktree.** This leaf's impl branch was created via `git worktree add` off `origin/main` to avoid touching the dirty state on the maintainer's main checkout (which carries unstaged SDLC-skill-migration deletes and a divergent `plan.yaml`). The implementer should plan to work in the worktree, not the main checkout, until the dirty state is resolved (which is out of scope for this leaf). Flag for implementer — surface in PR description.
