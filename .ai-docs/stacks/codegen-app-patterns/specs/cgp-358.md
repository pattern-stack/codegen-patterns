# CGP-358 — Emit service-layer composition methods for per-entity `relationships:` block

## TL;DR

Close the gap named by the CGP-62 audit: today the templates emit cross-entity access via Drizzle `with: { ... }` joins (the extension path), but the architecture says generated code MUST compose via single-table repo calls (the core contract). Replace the `include?: <Class>With` shape with paginated FK-traversal methods on **both** template pipelines.

**Recommendation: split this issue into three sub-leafs (see "Scope split" below) before implementation.** One PR for this surface is too large to review cleanly and the three sub-leafs have different risk profiles.

## Architectural rule (verbatim from CGP-62)

> Cross-entity access in generated code MUST go through service-layer composition via FK + single-table repo calls. No SQL JOINs at the service layer. Drizzle `relations()` consts continue to be emitted as table metadata, but generated service methods MUST NOT use Drizzle's `with: { ... }` query helper.

Two queries, no JOIN. `relations()` const stays as opt-in extension metadata for hand-written ad-hoc queries.

## Scope split (recommendation to planner)

The issue body describes three independent changes that should land as three PRs:

1. **CGP-358a — Clean-pipeline: emit composition use-cases; remove `include?` from repo + query surface.** Largest. Touches clean architecture only. Removes a public API shape (`<Class>With`, `findByIdWithRelations`, `findAllWithRelations`, eager-loaded readonly fields on the domain class). Adds N new use-case files per entity (one per relationship). Baseline snapshot churn is heavy. **Branch: `dug/cgp-358a-clean-pipeline-relationships`**
2. **CGP-358b — Clean-Lite-PS: emit composition methods on the service; bidirectional `relations()` const.** Closes the larger gap. Pure additive on a pipeline that currently emits *nothing* relationship-driven on the service/repo. Lower regression risk because there's no existing shape to break. Also extends the clean-lite-ps `relations()` emission to include `many()` calls (currently belongs_to / `one()` only). **Branch: `dug/cgp-358b-clean-lite-ps-relationships`**
3. **CGP-358c — Smoke fixture coverage + assertions.** New scenario fixtures under `test/smoke/fixtures/relationship-scenario/` and a `--scenario relationship` flag on `test/smoke/run-smoke.ts`. Asserts the new emissions against a CRM-shaped entity set. Independent of 358a/358b in code but must merge **after** them to assert real output. **Branch: `dug/cgp-358c-relationship-smoke`**

The remainder of this spec covers all three. If the planner declines the split, the implementer can still use this spec as a single PR strategy — but expect ~25 changed files and a fully-regenerated baseline.

## Architecture: what the templates must emit

### Inverse `has_many` / `has_one` (target side)

For an entity that declares `relationships.<name>: { type: has_many, target: <other>, foreign_key: <fk> }`, emit on that entity (the declaring side, which is also the "target" of the inverse FK):

```typescript
// account.yaml: relationships.contacts: { type: has_many, target: contact, foreign_key: account_id }
// → AccountService.contacts(id, opts) calls ContactRepository.findByAccountId
async contacts(accountId: string, opts?: { cursor?: string; limit?: number }): Promise<Contact[]> {
  return this.contactRepo.findByAccountId(accountId, opts);
}
```

Method name is the relationship key (`contacts`). FK column derives from the declared `foreign_key` (camelCase). Pagination opts always present on `has_many`; `has_one` returns scalar (`Promise<Contact | null>`) and takes no opts.

### `belongs_to` (declaring side)

```typescript
// contact.yaml: relationships.account: { type: belongs_to, target: account, foreign_key: account_id }
// → ContactService.account(contactId) — two single-table queries.
async account(contactId: string): Promise<Account | null> {
  const contact = await this.repository.findById(contactId);
  return contact ? this.accountRepo.findById(contact.accountId) : null;
}
```

Method name is the relationship key (`account`). Returns scalar (`Promise<Account | null>`). Two repo calls, no JOIN.

### Pagination convention

Use `{ cursor?: string; limit?: number }` per CGP-62 r4. The repo's `findBy<FK>Id` method gains a second `opts?` param of this shape, threaded through to a Drizzle `.where(...).limit(...)` chain. The cursor column is `id` (UUID, monotonic-enough for v1; revisit when ordering by `created_at` is needed). Grep first for any existing pagination shape on `queries:`-block methods; if present, adopt that. Today's `processedQueries` emit no pagination — the new shape is canonical.

#### Cross-entity module wiring

Composition methods on the clean pipeline need access to the *target's* repository. Options:

- **(A) Inject the target repo directly into the relationship query class.** Requires the target's module to `export` its repository token. The current module template already does this (line 64: `exports: [<%= repositoryToken %>, ...]`). The relationship query class's NestJS module must `imports: [<TargetEntity>Module]`. The `module.ejs.t` must add target modules to its `imports:` array when relationships exist with `existingTargets`.
- **(B) Defer cross-entity via the target's *service* (via a query use-case there).** More layers, no benefit. Reject.

Choose (A). The implementer must add the cross-module import wiring in `module.ejs.t`. The `processBelongsTo`/`processHasMany` already filter to `existingBelongsTo`/`existingHasMany` (entities whose YAML has been generated); use those. Two-pass generation already seeds entity files first (per CLAUDE.md baseline-test note), so by the time the second pass runs, target modules exist.

```typescript
// Domain layer — repository interface (clean pipeline)
// templates/entity/new/backend/domain/repository-interface.ejs.t
export interface IContactRepository {
  create(input: CreateContactInput, tx?: DrizzleTransaction): Promise<Contact>;
  findById(id: string): Promise<Contact | null>;             // include? removed
  findAll(): Promise<Contact[]>;                              // include? removed
  update(id: string, input: UpdateContactInput, tx?: DrizzleTransaction): Promise<Contact | null>;
  delete(id: string, tx?: DrizzleTransaction): Promise<Contact | null>;
  findByAccountId(id: string, opts?: { cursor?: string; limit?: number }): Promise<Contact[]>;
}

// Application layer — new composition query class (clean pipeline)
// templates/entity/new/backend/application/queries/relationships.queries.ts (emitted)
@Injectable()
export class AccountContactsQuery {
  constructor(
    @Inject(CONTACT_REPOSITORY) private readonly contactRepo: IContactRepository,
  ) {}
  async execute(accountId: string, opts?: { cursor?: string; limit?: number }): Promise<Contact[]> {
    return this.contactRepo.findByAccountId(accountId, opts);
  }
}

@Injectable()
export class ContactAccountQuery {
  constructor(
    @Inject(CONTACT_REPOSITORY) private readonly contactRepo: IContactRepository,
    @Inject(ACCOUNT_REPOSITORY) private readonly accountRepo: IAccountRepository,
  ) {}
  async execute(contactId: string): Promise<Account | null> {
    const contact = await this.contactRepo.findById(contactId);
    return contact ? this.accountRepo.findById(contact.accountId) : null;
  }
}

// Clean-Lite-PS service (in-class methods, no use-case split)
// templates/entity/new/clean-lite-ps/service.ejs.t (emitted)
@Injectable()
export class AccountService extends WithAnalytics(BaseService<AccountRepository, Account>) {
  constructor(
    protected override readonly repository: AccountRepository,
    private readonly contactRepo: ContactRepository,   // injected for has_many
  ) { super(repository); }

  async contacts(accountId: string, opts?: { cursor?: string; limit?: number }): Promise<Contact[]> {
    return this.contactRepo.findByAccountId(accountId, opts);
  }
}
```

## Out of scope

- **Junction association methods** (per issue body) — sibling leaf #357.
- **Removing the `relations()` const** — kept as opt-in extension.
- **Client-side composition codegen** — separate concern (Electric / generated frontend collections).
- **Cursor over non-`id` columns** — v1 cursor is `id`-only. Sort-stable cursor pagination (e.g., `created_at, id`) is a follow-up issue. File one if a fixture demands it.
- **Mutating methods on the inverse side** (e.g., `AccountService.attachContact`) — those are the junction pattern's territory; for plain `has_many` traversal, only the read side is in scope.

## Open questions (resolve before implementation)

1. **Single-file vs per-relationship use-case emission (clean pipeline)** — recommend single `relationships.queries.ts` file (matches `declarative-queries.ejs.t` shape). Confirm with implementer.
2. **Cursor column** — `id` for v1 (UUID, monotonic-enough). If a fixture surfaces ordering issues, file follow-up; do not block.
3. **`AccountService.contacts` collision with field names** — if an entity has a column named `contacts` (unlikely but possible), the method shadows nothing because methods and columns live in different namespaces, but YAML validation should warn. Out of scope for this leaf; file a validator-side follow-up if needed.
4. **`has_one` traversal** — issue body mentions but doesn't strongly require. Emit if simple (scalar `Promise<T | null>`, identical to `belongs_to` shape but inverse-side). Recommend implementing for symmetry; cost is negligible.
5. **Naming on self-referential `has_many`** — e.g., `account.yaml` has `relationships.children: { type: has_many, target: account, foreign_key: parent_account_id }`. The composition method needs the FK explicitly. Verify `processHasMany` picks up `foreign_key` correctly for self-refs.
