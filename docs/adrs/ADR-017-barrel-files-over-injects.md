# ADR-017 — Barrel Files over Hygen Injects

**Status:** Accepted
**Date:** 2026-04-13
**Owner:** Doug
**Related:** DOGFOOD-LOG.md #3, #4

## Context

Hygen's `inject: true` action modifies existing files by searching for a text anchor and splicing content around it. Codegen used this pattern to wire newly-generated entities into user-authored root files:

- `app.module.ts` — receive the `AccountsModule` import + array entry
- `packages/db/src/server/schema.ts` — receive the Drizzle table definition
- `constants/tokens.ts` — receive the `ACCOUNT_REPOSITORY` Symbol

Three hard failures surfaced during dogfood (#3, #4 in `DOGFOOD-LOG.md`):

1. **Silent corruption.** When a user's `app.module.ts` didn't contain the exact anchor comment Hygen expected (`// Codegen module imports`), Hygen didn't fail — it fell back to position 0 and inserted the import halfway up the file, above the real imports. The next `tsc` run broke with a baffling error, far removed from the codegen invocation.
2. **Hardcoded cwd paths.** Some inject templates wrote `to: './constants/tokens.ts'`. Real projects put `tokens.ts` under `shared/` or `infrastructure/constants/`. The inject failed with `Cannot inject to ./constants/tokens.ts: doesn't exist`, which Hygen treated as a non-fatal warning — generation "succeeded" and the user noticed hours later when runtime DI threw.
3. **Two systems of truth.** Every new entity required updating a user file *and* the entity YAML. Rolling back a generation (delete YAML + regenerate) left the user file still wired to a module that no longer exists.

The root cause isn't the anchor mechanism — it's the category mistake. Codegen was mutating files that the user also edits. Any such file is an arena for merge conflicts, drift, and subtle anchor-matching bugs.

## Decision

Codegen never mutates user-authored files. Instead, codegen owns two barrel files that aggregate the full entity set:

```
<paths.generated>/modules.ts  — exports GENERATED_MODULES (array of every entity's NestJS module)
<paths.generated>/schema.ts   — re-exports every entity's Drizzle entity/schema file
```

`paths.generated` is configurable in `codegen.config.yaml`, defaulting to `src/generated`.

Users wire these up **exactly once**, in code they author:

```ts
// src/app.module.ts — user file, written once
import { Module } from '@nestjs/common';
import { DatabaseModule } from './database/database.module';
import { GENERATED_MODULES } from './generated/modules';

@Module({
  imports: [DatabaseModule, ...GENERATED_MODULES],
})
export class AppModule {}
```

```ts
// src/schema.ts — user file, written once
export * from './generated/schema';
```

Every `codegen entity new` and `codegen entity new --all` invocation **fully regenerates** both barrels by scanning every `.yaml` file in the entities directory. The regeneration is total and deterministic — no state tracking, no incremental mutation, no anchor matching. Delete an entity YAML, run codegen, and the barrel no longer references it. The barrels are sorted alphabetically by entity name so the diff on each run is stable and small.

Both barrel files carry an `AUTO-GENERATED by @pattern-stack/codegen. Do not edit.` header.

## Consequences

### Positive

- **No silent corruption.** The only files codegen writes are fully owned by codegen. Malformed output is impossible because the file is built from scratch every run.
- **No anchor drift.** There is no anchor to match. Format of the user's `app.module.ts` is irrelevant — codegen never opens it.
- **Cheap rollback.** Remove a YAML, re-run, the module disappears from the barrel. No orphan imports.
- **Path-configurable.** The `paths.generated` config entry lets consumers with unusual layouts (monorepo, `apps/api/src`, etc.) pick a location that fits their tsconfig path aliases.
- **Deterministic builds.** Same input YAMLs always produce byte-identical barrels. CI diffs are meaningful.

### Negative

- **One-time migration cost.** Existing users who had Hygen-injected `app.module.ts` files must delete their injected lines and replace them with the two-line import + spread. `docs/GETTING-STARTED.md` shows the exact pattern.
- **Barrels don't honor every `locations:` override.** The barrel currently emits paths derived from default `locations:` entries. Projects that remap `backendModules` or `backendDrizzle` via `codegen.config.yaml` won't see the remapped paths reflected in the barrel. This is a known limitation; see the TODO in `src/cli/shared/barrel-generator.ts`. The `paths.backend_src` override is honored.
- **Less flexibility for one-off modules.** If a user wants to wire up a module conditionally (e.g. behind a feature flag), they can't use `GENERATED_MODULES` directly; they'd spread selectively or wire that module by hand outside the barrel. Acceptable — one-off overrides are rare and the escape hatch is trivial (import the module directly from its file).

### Neutral

- The `GENERATED_MODULES` array is typed `as const` so TypeScript preserves the exact module class tuple at the type level. Consumers who want a structural type can widen with `[...typeof GENERATED_MODULES]`.

## Alternatives Considered

### 1. Loud-failing Hygen injects

Make the inject templates throw when their anchor isn't found, rather than silently splicing at position 0. This fixes the silent-corruption symptom but not the underlying problem — the user's file still has to contain a specific comment string, and every path we inject into is a separate failure surface. Rejected: treats the symptom.

### 2. AST-based rewriting via ts-morph

Parse `app.module.ts` with ts-morph, find the `@Module` decorator, splice into `imports: []`. Technically robust but:
- Heavy runtime dependency (ts-morph is large).
- The user's file can still fight back: wrong variable name, destructured imports, conditional module lists, custom decorators — each is a special case.
- Still violates the "don't mutate user files" principle, just with a fancier lockpick.

Rejected: solves the wrong problem at a high complexity cost.

### 3. User writes nothing — codegen owns `app.module.ts` entirely

Go further: have codegen generate `app.module.ts` as a whole. Rejected because `app.module.ts` is typically where users wire in manually-authored modules (`AuthModule`, `HealthModule`, third-party integrations). Forcing those into config would make codegen responsible for a much larger surface than entity wiring.

The barrel approach is the Goldilocks option: codegen owns exactly the aggregation of entity-derived modules and schemas, nothing more.

## Migration

For existing codegen consumers:

1. **Remove** previously-injected imports and array entries from `app.module.ts`. Any line that mentions a generated `*Module` from `./modules/...` or `./infrastructure/modules/...` is a candidate for deletion.
2. **Add** the two-line wire-up shown above — one import, one spread into the `imports:` array.
3. **Remove** the Drizzle table definitions previously injected into the monolithic `packages/db/src/server/schema.ts` — they're now read through the `<generated>/schema.ts` barrel. If `schema.ts` was user-authored, keep that file but have it `export * from './generated/schema';`.
4. **Run** `codegen entity new --all` once to produce the barrels.

Old inject templates have been removed from the distribution (see the PR that lands this ADR), so there's nothing to disable — just run `codegen` and the new barrels appear.
