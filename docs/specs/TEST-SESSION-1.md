## 1. Goals & Non-Goals

**Goals:**
- Prove jobs + events run end-to-end against a real Postgres instance from a consumer app generated by this codegen
- Exercise the six stress scenarios that unit tests can't cover (horizontal scale, cascade cancel, dedupe, step memoization across crash, tenant isolation, event fan-out)
- Build a minimal server-rendered viewer (`/admin/jobs`, `/admin/events`) sufficient to observe and debug runs during the session
- File any bugs discovered; establish a regression baseline

**Non-goals:**
- Performance benchmarking
- Event-to-Job Bridge (ADR-023, Phase 2) — jobs and events are dispatched independently in this session
- Frontend polish, authentication on the viewer
- Subsystems not yet shipped (cache, storage)

### `events/*.yaml` Files

Event YAML uses `type:` (not `name:`) top-level. `payload:` is a **record** keyed by snake_case field name. Payload field types are narrower than entity field types: `uuid | string | number | boolean | date | json` — no `string[]`, no `integer`, no `timestamp`/`datetime`. When `direction: change`, `aggregate:` is required; `direction: outbound` uses `destination:` (not `aggregate:`).

**`events/contact_created.yaml`**
```yaml
type: contact_created
direction: change
aggregate: contact
version: 1
payload:
  contact_id: { type: string }
  email:      { type: string }
```

**`events/contact_updated.yaml`**
```yaml
type: contact_updated
direction: change
aggregate: contact
version: 1
payload:
  contact_id:     { type: string }
  changed_fields: { type: json }   # array types aren't in the payload enum — use json
```

**`events/deal_created.yaml`**
```yaml
type: deal_created
direction: change
aggregate: deal
version: 1
payload:
  deal_id:    { type: uuid }
  contact_id: { type: uuid }
```

**`events/deal_stage_changed.yaml`**
```yaml
type: deal_stage_changed
direction: change
aggregate: deal
version: 1
payload:
  deal_id:    { type: uuid }
  from_stage: { type: string }
  to_stage:   { type: string }
```

**`events/crm_sync_requested.yaml`**
```yaml
type: crm_sync_requested
direction: outbound
destination: crm                   # `aggregate:` is only valid with `direction: change`
version: 1
payload:
  deal_id:      { type: uuid }
  crm_provider: { type: string }
```

### `@JobHandler` Classes

**`enrich-contact.job.ts`** — exercises `ctx.step` memoization + retry

```typescript
@JobHandler({ type: 'enrich_contact', pool: 'default', retry: { attempts: 3, backoff: 'exponential' } })
export class EnrichContactJob extends BaseJobHandler<{ contactId: string }> {
  async handle(ctx: JobContext<{ contactId: string }>): Promise<void> {
    // Step 1: fetch enrichment data (idempotent — memoized on re-run)
    const enrichmentData = await ctx.step('fetch_enrichment', async () => {
      // simulates external API call; throw on first attempt to exercise retry
      return { company: 'Acme Corp', linkedin: 'https://linkedin.com/...' };
    });

    // Step 2: persist result (separate step — only re-runs if step 1 completed)
    await ctx.step('persist_enrichment', async () => {
      // update contact record
    });
  }
}
```

**`sync-deal.job.ts`** — exercises `ctx.spawnChild`, scope binding, `ScopeRef`

```typescript
@JobHandler({
  type: 'sync_deal',
  pool: 'default',
  scope: { entityType: 'deal' },   // scope_entity_type = 'deal'
})
export class SyncDealJob extends BaseJobHandler<{ dealId: string }> {
  async handle(ctx: JobContext<{ dealId: string }>): Promise<void> {
    const lines = await ctx.step('fetch_line_items', async () => {
      return [{ id: 'line-1' }, { id: 'line-2' }];
    });

    // Spawn one child job per line item; parent waits (terminate policy)
    for (const line of lines) {
      await ctx.spawnChild('sync_line_item', {
        input: { lineId: line.id, dealId: ctx.input.dealId },
        parentClosePolicy: 'terminate',
      });
    }
  }
}

@JobHandler({ type: 'sync_line_item', pool: 'default' })
export class SyncLineItemJob extends BaseJobHandler<{ lineId: string; dealId: string }> {
  async handle(ctx: JobContext<{ lineId: string; dealId: string }>): Promise<void> {
    await ctx.step('sync', async () => { /* push to CRM */ });
  }
}
```

**`notify-crm.job.ts`** — exercises dedupe + `replace` collision mode

```typescript
@JobHandler({
  type: 'notify_crm',
  pool: 'default',
  concurrency: { mode: 'replace' },  // replace any pending run for same dedupe key
  dedupe: { key: 'deal_id', windowSeconds: 30 },
})
export class NotifyCrmJob extends BaseJobHandler<{ dealId: string }> {
  async handle(ctx: JobContext<{ dealId: string }>): Promise<void> {
    await ctx.step('push_to_crm', async () => { /* notify external CRM */ });
  }
}
```

### Event Subscribers

After EVT-7, generated use-cases call `TypedEventBus.publish()` with `{ tx }`. Subscribers register via `IEventBus.subscribe()` (the drain side). For Phase 1 (no Event-to-Job Bridge yet), wire a plain subscriber that logs received events:

```typescript
// src/subscribers/deal-stage.subscriber.ts
@Injectable()
export class DealStageSubscriber implements OnApplicationBootstrap {
  constructor(@Inject(TYPED_EVENT_BUS) private readonly bus: TypedEventBus) {}

  onApplicationBootstrap() {
    this.bus.subscribe('deal_stage_changed', async (event) => {
      console.log('[subscriber] deal stage changed:', event.payload);
      // In Phase 2 (ADR-023), this would dispatch a NotifyCrmJob
    });
  }
}
```

Note: The drain (actually calling subscribers) requires the event bus `processBatch()` to run. In Phase 1, wire a `setInterval` or a dedicated drain module that calls `processBatch()` on the `events_change` pool every 2 seconds. This simulates what the Event-to-Job Bridge will own in Phase 2.

## 4. Stress-Test Scenarios

### S1: Horizontal Scale — Two Workers, One Pool

**Setup:** Start two separate Node processes, both running `JobWorkerModule.forRoot` against the same Postgres, same pool (`default`).

**Steps:**
1. Dispatch 20 `enrich_contact` jobs in rapid succession
2. Observe both workers claim jobs (`FOR UPDATE SKIP LOCKED`)
3. Verify no duplicate claims (each `job_run.id` claimed by exactly one worker)
4. Kill one worker mid-run; verify stale-claim sweeper on the surviving worker picks up abandoned claims within the configured sweep interval

**Pass:** All 20 jobs complete exactly once. No `claimed_by` conflicts in `job_run` table.

### S2: Cascade Cancel — Parent with Children

**Setup:** Use `sync_deal` handler (spawns 2 child `sync_line_item` jobs per deal).

**Steps:**
1. Dispatch `sync_deal` job; let it spawn children (observe `parent_run_id` set on children)
2. Before children complete, POST to `/admin/jobs/:parentRunId/cancel`
3. Verify `parent_close_policy: terminate` causes children to be marked `canceled`
4. Verify parent reaches `canceled` state

**Pass:** Parent + all children → `canceled`. No orphaned `running` children.

### S3: Dedupe Collapse Inside Window

**Setup:** Use `notify_crm` handler (dedupe key `deal_id`, window 30s, collision mode `replace`).

**Steps:**
1. Dispatch `notify_crm` for `deal-123` — observe run enters `pending`
2. Dispatch again for same `deal_id` within 30s — observe prior pending run replaced
3. Verify only one `running` run exists for the deal at any time
4. Dispatch after 30s — should create a fresh run (no dedup)

**Pass:** At most one active run per `deal_id` inside the window. Second dispatch replaces, not queues.

### S4: Step Memoization Across Simulated Crash

**Setup:** Use `enrich_contact` handler with 2 steps.

**Steps:**
1. Modify `EnrichContactJob.handle` to throw after step 1 completes (simulate crash)
2. Dispatch job; observe `job_step` row for `fetch_enrichment` created with `status: completed`
3. Job run fails; re-dispatch same job using replay mode `last_step`
4. Verify step 1 is NOT re-executed (reads memoized result from `job_step`)
5. Step 2 (`persist_enrichment`) executes and completes

**Pass:** `job_step.fetch_enrichment.execution_count = 1` after 2 run attempts.

### S5: Tenant Isolation (optional — requires `multi_tenant: true`)

**Setup:** Re-scaffold with `jobs.multi_tenant: true` + `events.multi_tenant: true`, apply Atlas migration.

**Steps:**
1. Dispatch jobs for tenant A and tenant B in the same pool
2. Start worker configured for tenant A only
3. Verify worker only claims jobs where `tenant_id = tenantA`
4. Verify tenant B jobs remain `pending`

**Pass:** No cross-tenant claims. `MissingTenantIdError` thrown if dispatched without `tenant_id`.

### S6: Event Fan-Out Across Direction Pools

**Setup:** Two subscribers on `deal_stage_changed` (direction: `change`, pool: `events_change`).

**Steps:**
1. Create a deal, advance its stage via the generated use-case
2. Verify `domain_events` row inserted with `direction: change`, `pool: events_change`
3. Run drain cycle (processBatch on `events_change`)
4. Verify both subscribers receive the event exactly once

**Pass:** Both subscribers fire. `domain_events.status` → `processed`. No double-delivery on re-drain.

### S7: Atlas Migration Round-Trip

**Steps:**
1. Start with `multi_tenant: false` — apply base migration, seed data, verify jobs + events work
2. Update `codegen.config.yaml`: `jobs.multi_tenant: true`, `events.multi_tenant: true`
3. Run `atlas migrate diff` — verify diff adds `tenant_id` columns + index
4. Run `atlas migrate apply` — verify existing rows unchanged, new rows require `tenant_id`
5. Boot app, re-run S1 (horizontal scale) with tenant context

**Pass:** Migration applies cleanly. Pre-migration data survives. App boots without schema errors.

## 5. Viewer Design

### Architecture Decision

The viewer is a hand-written NestJS `AdminModule` inside the demo app. It is NOT shipped from codegen templates. Rationale: the viewer is a debugging tool for this test session; shipping it from templates would mean adding template infrastructure for a feature that doesn't belong in the codegen product yet (that's ADR-026 observability, Phase 4). Building it by hand in the demo app is ~1 hour of work and keeps templates clean.

Styling: plain inline CSS, no build step. NestJS `res.send()` returning inline HTML — no template engine setup required. Legible, not pretty.

### Routes

| Route | Description |
|-------|-------------|
| `GET /admin/` | Index: counts + links for jobs and events |
| `GET /admin/jobs` | Job runs list — filter: status, pool, job_type, scope_entity_type |
| `GET /admin/jobs/:runId` | Run detail: input, status, started/completed timestamps, step timeline (ordered `job_step` rows), child run tree (recursive `parent_run_id` query), parent link |
| `POST /admin/jobs/:runId/cancel` | Cancel a run (calls `IJobRunService.cancel()`); redirect back to run detail |
| `POST /admin/jobs/:runId/replay` | Replay a completed/failed run; redirect to new run detail |
| `GET /admin/events` | `domain_events` list — filter: pool, direction, status, aggregate_type |
| `GET /admin/events/:eventId` | Event detail: full payload, occurred_at, status, tenant_id if set; "triggered jobs" section (static note: bridge not yet wired) |

### Implementation Notes

- Use Drizzle directly in `AdminController` — no extra service layer. This is a dev tool.
- Job run list query: `SELECT * FROM job_run ORDER BY created_at DESC LIMIT 50` with optional WHERE filters via query params
- Step timeline: `SELECT * FROM job_step WHERE run_id = $1 ORDER BY started_at ASC`
- Child tree: single recursive CTE or two-level query (Phase 1 trees are shallow — children don't spawn grandchildren in our fixtures)
- Pagination: offset-based, 50 rows per page. No infinite scroll.
- Cancel/replay: minimal `<form method="POST">` — no AJAX

### Sample HTML Structure (pseudocode)

```html
<!-- /admin/jobs/:runId -->
<h2>Run: {runId}</h2>
<dl>
  <dt>Type</dt><dd>{job_type}</dd>
  <dt>Status</dt><dd>{status}</dd>
  <dt>Pool</dt><dd>{pool}</dd>
  <dt>Started</dt><dd>{started_at}</dd>
  <dt>Completed</dt><dd>{completed_at ?? '—'}</dd>
</dl>

<h3>Steps</h3>
<table>
  <tr><th>Step ID</th><th>Status</th><th>Started</th><th>Duration</th><th>Output</th></tr>
  {#each steps}
  <tr><td>{step_id}</td><td>{status}</td><td>{started_at}</td><td>{ms}</td><td><pre>{output}</pre></td></tr>
  {/each}
</table>

<h3>Child Runs</h3>
{#if children.length}
<ul>{#each children}<li><a href="/admin/jobs/{run_id}">{job_type} — {status}</a></li>{/each}</ul>
{:else}<p>None</p>{/if}

<form method="POST" action="/admin/jobs/{runId}/cancel"><button>Cancel</button></form>
<form method="POST" action="/admin/jobs/{runId}/replay"><button>Replay</button></form>
```

### Blocking vs. Follow-Up Failures

**Session-blocking** (stop and fix before continuing):
- App won't boot (injection errors, schema mismatch)
- Worker claims same job_run twice
- Atlas migration corrupts existing rows
- Cascade cancel leaves children in `running` indefinitely

**File as follow-up** (continue session):
- Viewer UI bugs (wrong timestamp format, missing field)
- Edge case in dedupe (boundary condition at window edge)
- Slow drain cycle
- Missing index causing slow admin query

## 7. Risk Areas to Probe

### Strong Unit Coverage, No Integration Test

| Area | Risk | How to Probe |
|------|------|-------------|
| Stale-claim sweeper | Sweep interval, SKIP LOCKED behavior against real Postgres | S1: kill worker mid-run, measure recovery time |
| Boot-time hash-gated upsert under concurrent instances | Two workers booting simultaneously could race on `ON CONFLICT` | Start both workers at exactly the same time; verify `job` table has exactly one row per handler type |
| Scaffold-time schema conditional | `{% if multi_tenant %}` in schema template must render both variants correctly for real Atlas | S7: generate both variants, diff + apply each |
| `parent_close_policy: terminate` propagation | State machine transitions tested in isolation, not with real concurrent workers | S2: ensure child cancel happens even if child is mid-step |
| Step memoization re-entry | `ctx.step()` reads existing row instead of re-executing — tested with mock store | S4: verify against real Drizzle backend |
| `dedupe` window boundary | Window is checked at dispatch time; clock skew or slow clock can cause edge cases | S3: test exactly at window boundary |

### Cross-Subsystem Interactions Not Yet Exercised

| Interaction | Risk |
|-------------|------|
| Jobs + Events in same AppModule | Shared Drizzle client — potential connection pool exhaustion if both subsystems hold long transactions |
| Event drain using reserved job pools | The three `events_*` pools are reserved; if a JobWorker accidentally picks them up, fan-out breaks. Test that user handlers cannot be registered against reserved pools. |
| TypedEventBus publish inside a job step | Not a standard flow (events usually published from use-cases), but may occur. Verify `tx` handling is correct when called from `ctx.step()`. |
| Atlas migration with existing `domain_events` rows | Multi-tenant migration adds `tenant_id NOT NULL`? Confirm column is nullable or has default. |
